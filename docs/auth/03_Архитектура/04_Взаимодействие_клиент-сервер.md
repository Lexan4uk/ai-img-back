# 04. Взаимодействие клиент-сервер

## Общая схема

В auth-подсистеме участвуют **три стороны**:

```
┌──────────────────────┐       ┌──────────────────────┐       ┌──────────────────┐
│    Desktop Client    │       │   Основной бэкенд    │       │   Auth-сервер    │
│  (Compose / Kotlin)  │ HTTP  │   (Spring Boot)      │ HTTP  │  (Spring Boot)   │
│                      │◄─────►│                      │◄─────►│                  │
│  auth-client-utils   │ :8080 │  AuthController      │ :9090 │  AuthController  │
│  AuthInterceptor     │       │  JwtAuthFilter       │       │  TokenService    │
│  TokenManager        │       │  JJWT (локально)     │       │  AuthService     │
└──────────────────────┘       └──────────────────────┘       └──────────────────┘
```

**Клиент** никогда не обращается к auth-серверу напрямую. Все запросы идут через основной бэкенд, который **проксирует** auth-операции и **валидирует** токены для бизнес-запросов.

---

## Три типа HTTP-запросов

### 1. Auth-запросы (проксируемые)

```
Клиент → Основной бэкенд → Auth-сервер → ответ → Основной бэкенд → Клиент
```

Регистрация, логин, refresh, logout, смена пароля. Основной бэкенд пробрасывает тело запроса на auth-сервер и возвращает ответ клиенту.

### 2. Валидация токена (локальная)

```
Клиент → Основной бэкенд (JwtAuthFilter → JJWT локально) → бизнес-логика
```

JwtAuthFilter извлекает Bearer-токен, валидирует JWT локально через JJWT (подпись + exp). HTTP-вызов к auth-серверу не нужен. Клиент не знает о валидации.

### 3. Бизнес-запросы (с Bearer-токеном)

```
Клиент (Bearer) → Основной бэкенд (JwtAuthFilter → OK) → Controller → Service → ответ
```

Обычные запросы к API. AuthInterceptor на клиенте автоматически добавляет `Authorization: Bearer <accessToken>`.

---

## Авторизация

### До auth-модуля (текущее состояние)

```
Клиент ──[Header: UserId: 2]──► Сервер
```

### После auth-модуля

```
Клиент ──[Header: Authorization: Bearer eyJhb...]──► Сервер
```

| Аспект          | До                                  | После                                           |
| --------------- | ----------------------------------- | ------------------------------------------------ |
| Идентификация   | `UserId` в заголовке (ручная)       | JWT accessToken (автоматическая)                 |
| Подстановка     | OkHttp Interceptor в `ClientUtil`   | `AuthInterceptor` в `auth-client-utils`          |
| Чтение сервером | `@RequestHeader("UserId")`          | `@AuthenticationPrincipal AuthUser`              |
| Безопасность    | нет (можно подставить любой userId) | подпись JWT, локальная валидация (JJWT)          |
| Обновление      | не нужно                            | auto-refresh при 401 (AuthInterceptor)           |

## Потоки auth-запросов

### Регистрация

```
Клиент                          Бэкенд (:8080)              Auth-сервер (:9090)
  │                                │                              │
  │  POST /auth/register           │                              │
  │  { email, password,            │                              │
  │    displayName }               │                              │
  │ ──────────────────────────►    │                              │
  │                                │  POST /register              │
  │                                │  { email, password,          │
  │                                │    displayName }             │
  │                                │ ────────────────────────►    │
  │                                │                              │── BCrypt(password)
  │                                │                              │── INSERT auth_users
  │                                │                              │── generatePair()
  │                                │  { accessToken,              │
  │                                │    refreshToken }            │
  │                                │ ◄────────────────────────    │
  │  { accessToken,                │                              │
  │    refreshToken }              │                              │
  │ ◄──────────────────────────    │                              │
  │                                │                              │
  │── saveTokens(access, refresh)  │                              │
  │── decode(access) → UserClaims  │                              │
  │── UI: Галерея                  │                              │
```

### Логин

```
Клиент                          Бэкенд (:8080)              Auth-сервер (:9090)
  │                                │                              │
  │  POST /auth/login              │                              │
  │  { email, password }           │                              │
  │ ──────────────────────────►    │                              │
  │                                │  POST /login                 │
  │                                │  { email, password }         │
  │                                │ ────────────────────────►    │
  │                                │                              │── SELECT по email
  │                                │                              │── BCrypt.matches()
  │                                │  { accessToken,              │
  │                                │    refreshToken }            │
  │                                │ ◄────────────────────────    │
  │  { accessToken,                │                              │
  │    refreshToken }              │                              │
  │ ◄──────────────────────────    │                              │
  │                                │                              │
  │── saveTokens(access, refresh)  │                              │
  │── decode(access) → UserClaims  │                              │
  │── UI: Галерея                  │                              │
```

### Бизнес-запрос (с auto-refresh)

```
Клиент                          Бэкенд (:8080)              Auth-сервер (:9090)
  │                                │                              │
  │  GET /image-types              │                              │
  │  Authorization: Bearer <AT>    │                              │
  │ ──────────────────────────►    │                              │
  │                                │── JwtAuthFilter              │
  │                                │   JJWT: verify(signingKey)  │
  │                                │   check exp                 │
  │                                │   extract claims → AuthUser │
  │                                │   SecurityContext ← AuthUser│
  │                                │                              │
  │                                │── @AuthenticationPrincipal  │
  │                                │   → AuthUser → userId       │
  │                                │── Controller → Service       │
  │  200 OK [результат]            │                              │
  │ ◄──────────────────────────    │                              │
```

**При 401 (токен истёк):**

```
Клиент                          Бэкенд (:8080)              Auth-сервер (:9090)
  │                                │                              │
  │  GET /image-types              │                              │
  │  Authorization: Bearer <AT>    │                              │
  │ ──────────────────────────►    │                              │
  │  ◄── 401 Unauthorized          │                              │
  │                                │                              │
  │── AuthInterceptor:             │                              │
  │   refreshToken есть?           │                              │
  │   └── Да:                      │                              │
  │                                │                              │
  │  POST /auth/refresh            │                              │
  │  { refreshToken }              │                              │
  │ ──────────────────────────►    │                              │
  │                                │  POST /refresh               │
  │                                │  { refreshToken }            │
  │                                │ ────────────────────────►    │
  │                                │  { accessToken }             │
  │                                │ ◄────────────────────────    │
  │  { accessToken }               │                              │
  │ ◄──────────────────────────    │                              │
  │                                │                              │
  │── saveAccessToken(newAT)       │                              │
  │                                │                              │
  │  GET /image-types              │                              │
  │  Authorization: Bearer <newAT> │                              │
  │ ──────────────────────────►    │                              │
  │  200 OK [результат]            │                              │
  │ ◄──────────────────────────    │                              │
```

Весь auto-refresh прозрачен для вызывающего кода — `AuthInterceptor` обрабатывает 401 автоматически.

### Refresh

```
Клиент                          Бэкенд (:8080)              Auth-сервер (:9090)
  │                                │                              │
  │  POST /auth/refresh            │                              │
  │  { refreshToken }              │                              │
  │ ──────────────────────────►    │                              │
  │                                │  POST /refresh               │
  │                                │  { refreshToken }            │
  │                                │ ────────────────────────►    │
  │                                │                              │── validate(RT)
  │                                │                              │── SELECT auth_users
  │                                │                              │── generateAccess()
  │                                │  { accessToken }             │
  │                                │ ◄────────────────────────    │
  │  { accessToken }               │                              │
  │ ◄──────────────────────────    │                              │
```

При refresh auth-сервер читает **актуальные данные** пользователя из БД и записывает их в новый accessToken (displayName, roles могли измениться).

### Logout

```
Клиент                          Бэкенд (:8080)              Auth-сервер (:9090)
  │                                │                              │
  │  POST /auth/logout             │                              │
  │  { accessToken, refreshToken } │                              │
  │ ──────────────────────────►    │                              │
  │                                │  POST /logout                │
  │                                │  { accessToken,              │
  │                                │    refreshToken }            │
  │                                │ ────────────────────────►    │
  │                                │                              │── blacklist(RT.jti)
  │                                │  204 No Content              │
  │                                │ ◄────────────────────────    │
  │  204 No Content                │                              │
  │ ◄──────────────────────────    │                              │
  │                                │                              │
  │── clearTokens()                │                              │
  │── UI: экран Логина             │                              │
```

### Смена пароля

```
Клиент                          Бэкенд (:8080)              Auth-сервер (:9090)
  │                                │                              │
  │  POST /auth/change-password    │                              │
  │  Authorization: Bearer <AT>    │                              │
  │  { currentPassword,            │                              │
  │    newPassword }               │                              │
  │ ──────────────────────────►    │                              │
  │                                │── JwtAuthFilter → OK         │
  │                                │── извлечь userId из claims   │
  │                                │                              │
  │                                │  POST /change-password       │
  │                                │  { userId, currentPassword,  │
  │                                │    newPassword }             │
  │                                │ ────────────────────────►    │
  │                                │                              │── BCrypt.matches(current)
  │                                │                              │── BCrypt.encode(new)
  │                                │                              │── UPDATE password_hash
  │                                │  204 No Content              │
  │                                │ ◄────────────────────────    │
  │  204 No Content                │                              │
  │ ◄──────────────────────────    │                              │
```

Обратить внимание: клиент отправляет `{ currentPassword, newPassword }` **без userId** — бэкенд извлекает `userId` из JWT-токена. Это защита от подмены чужого userId.

## Формат данных

### Запросы

| Тип  | Content-Type     | Тело                      |
| ---- | ---------------- | ------------------------- |
| POST | application/json | JSON-объект               |
| GET  | —                | параметры в URL (бизнес)  |

Все auth-запросы — POST с JSON-телом.

### Ответы (auth)

| Код | Формат | Когда                                         |
| --- | ------ | --------------------------------------------- |
| 200 | JSON   | Успешный login / register / refresh |
| 204 | —      | Успешный logout / change-password              |
| 400 | JSON   | Ошибка валидации (пустой email, короткий пароль) |
| 401 | JSON   | Неверный пароль / невалидный токен             |
| 409 | JSON   | Email уже зарегистрирован                      |
| 500 | JSON   | Внутренняя ошибка                              |

### Формат ошибки

```json
{ "error": "текст ошибки" }
```

Единый формат для auth-сервера и основного бэкенда.

## Полный перечень auth API

Все auth-эндпоинты основного бэкенда (проксируемые на auth-сервер):

| Метод | URL бэкенда (:8080)    | URL auth-сервера (:9090) | Тело запроса                        | Ответ (успех)              | Auth нужен |
| ----- | ---------------------- | ------------------------ | ----------------------------------- | -------------------------- | ---------- |
| POST  | `/auth/register`       | `/register`              | `{ email, password, displayName }`  | `{ accessToken, refreshToken }` | нет   |
| POST  | `/auth/login`          | `/login`                 | `{ email, password }`               | `{ accessToken, refreshToken }` | нет   |
| POST  | `/auth/refresh`        | `/refresh`               | `{ refreshToken }`                  | `{ accessToken }`          | нет        |
| POST  | `/auth/logout`         | `/logout`                | `{ accessToken, refreshToken }`     | `204 No Content`           | да         |
| POST  | `/auth/change-password`| `/change-password`       | `{ currentPassword, newPassword }`  | `204 No Content`           | да         |

> **`/auth/validate` — не проксируется, не доступен клиенту.**
> Эндпоинт `POST /validate` существует только на auth-сервере (:9090) и используется им **внутренне** — при refresh и logout для проверки jti по blacklist. Основной бэкенд валидирует JWT **локально** через JJWT (подпись + exp), поэтому HTTP-вызов к `/validate` ему не нужен. Клиент не знает о существовании этого эндпоинта.

### Маппинг URL

Основной бэкенд добавляет префикс `/auth/` к клиентским запросам. Auth-сервер принимает запросы **без** этого префикса:

```
Клиент: POST /auth/login  →  Бэкенд: прокси  →  Auth-сервер: POST /login
```

### Защищённые эндпоинты

Для `logout` и `change-password` основной бэкенд **проверяет Bearer-токен** перед проксированием. Для `register`, `login`, `refresh` — проксирование без проверки.

## Поведение при сбоях

### Auth-сервер недоступен

| Операция      | Поведение                                              |
| ------------- | ------------------------------------------------------ |
| Логин         | 503 → клиент показывает «Сервис временно недоступен»   |
| Регистрация   | 503 → аналогично                                       |
| Refresh       | 503 → AuthInterceptor: onLoginRequired()               |
| Бизнес-запрос | JwtAuthFilter: локальная валидация → работает автономно  |
| Logout        | 503 → клиент всё равно clearTokens() + экран логина    |

Локальная валидация JWT (JJWT) позволяет бизнес-запросам работать **полностью автономно** — auth-сервер не нужен для проверки токенов. Бизнес-запросы продолжают работать даже при падении auth-сервера (пока accessToken не истёк).

### Таймауты

| Компонент                  | Таймаут | Retry |
| -------------------------- | ------- | ----- |
| Бэкенд → Auth-сервер      | 5 сек   | 1     |
| Клиент → Бэкенд (auth)    | 10 сек  | нет   |
| Клиент → Бэкенд (бизнес)  | зависит от операции | нет |

### Refresh не удался

```
AuthInterceptor: POST /auth/refresh
  ├── 200 { accessToken } → сохранить, повторить запрос
  ├── 401 → refreshToken истёк / в blacklist
  │         → clearTokens()
  │         → onLoginRequired()
  │         → UI: экран Логина
  └── 503 → auth-сервер недоступен
            → onLoginRequired()
```

### Потеря refreshToken

При отсутствии refreshToken (очистка Preferences, переустановка) — AuthInterceptor сразу вызывает `onLoginRequired()` без попытки refresh.

## Жизненный цикл подключения

```
Запуск клиента
    │
    ├── TokenManager.hasRefreshToken()?
    │   │
    │   ├── Да:
    │   │   POST /auth/refresh { refreshToken }
    │   │   ├── 200 → saveAccessToken() → decode() → Галерея
    │   │   └── 401/503 → экран Логина
    │   │
    │   └── Нет → экран Логина
    │
Экран Логина / Регистрации
    │
    ├── POST /auth/login (или /auth/register)
    │   ├── 200 → saveTokens() → decode() → Галерея
    │   ├── 401 → «Неверный email или пароль»
    │   └── 409 → «Email уже зарегистрирован»
    │
Работа с приложением
    │
    ├── Каждый бизнес-запрос:
    │   AuthInterceptor → Bearer <accessToken>
    │   └── 401 → auto-refresh → повтор
    │
Logout
    │
    ├── POST /auth/logout { accessToken, refreshToken }
    │   → clearTokens() → экран Логина
    │
Смена пароля
    │
    ├── POST /auth/change-password { currentPassword, newPassword }
    │   → 204 → «Пароль изменён» (токены остаются)
    │   → 401 → «Неверный текущий пароль»
    │
Закрытие приложения
    │
    ├── accessToken теряется (в памяти)
    ├── refreshToken сохраняется (Preferences)
    └── При следующем запуске → refresh → продолжение работы
```
