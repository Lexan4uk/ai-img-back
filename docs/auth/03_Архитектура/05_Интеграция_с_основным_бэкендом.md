# 05. Интеграция с основным бэкендом

## Назначение

Этот документ описывает, как основной бэкенд подключает авторизацию через Spring Security. Auth-модуль универсален — его механизмы (SecurityFilterChain, JWT-фильтр, аннотации, прокси-контроллер) применимы к любому контроллеру в проекте.

---

## Что добавляется в основной бэкенд

```
Основной бэкенд (Spring Boot)
│
├── config/
│   ├── ProdSecurityConfig       ← SecurityFilterChain (auth.enabled=true)
│   ├── DevSecurityConfig        ← Всё открыто (auth.enabled=false)
│   └── JwtAuthProperties        ← @ConfigurationProperties("jwt")
│
├── filter/
│   └── JwtAuthFilter            ← OncePerRequestFilter (валидация JWT)
│
├── model/
│   └── AuthUser                 ← Principal: id, email, displayName, role
│
├── proxy/
│   ├── AuthProxyController      ← Проксирование auth-запросов
│   ├── AuthProxyConfig          ← OkHttpClient bean
│   └── AuthServerProperties     ← @ConfigurationProperties("auth.server")
│
└── OkHttp клиент               ← HTTP к auth-серверу
```

Бизнес-контроллеры получают данные пользователя через `@AuthenticationPrincipal AuthUser` — стандартный механизм Spring Security.

## Spring Security конфигурация

### ProdSecurityConfig (`auth.enabled=true`)

Активный режим — JWT-фильтр проверяет все запросы.

```java
@Configuration
@ConditionalOnProperty(name = "auth.enabled", havingValue = "true")
public class ProdSecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http,
                                                    JwtAuthFilter jwtAuthFilter) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())
            .cors(cors -> cors.disable())
            .sessionManagement(session -> session.sessionCreationPolicy(STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/login", "/auth/register", "/auth/refresh").permitAll()
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}
```

### DevSecurityConfig (`auth.enabled=false`)

Режим разработки — авторизация отключена, все запросы разрешены.

```java
@Configuration
@ConditionalOnProperty(name = "auth.enabled", havingValue = "false", matchIfMissing = true)
public class DevSecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())
            .cors(cors -> cors.disable())
            .authorizeHttpRequests(auth -> auth.anyRequest().permitAll())
            .build();
    }
}
```

`matchIfMissing = true` — если `auth.enabled` не указан в properties, работает DevSecurityConfig (безопасный дефолт для разработки).

## JwtAuthFilter

### Принцип

`OncePerRequestFilter` из Spring Security. Перехватывает каждый запрос **до** контроллера. Валидирует JWT **локально** (JJWT + общий signing key) — HTTP-вызов к auth-серверу не нужен.

### Логика фильтра

| Шаг | Действие                                                  | При ошибке        |
| --- | --------------------------------------------------------- | ----------------- |
| 1   | Проверить путь — в `shouldNotFilter()`?                   | skip → контроллер |
| 2   | Прочитать заголовок `Authorization`                       | нет → 401         |
| 3   | Проверить префикс `Bearer `                               | нет → 401         |
| 4   | Извлечь токен (substring после `Bearer `)                 | пустой → 401      |
| 5   | Парсить JWT через JJWT (verifyWith + signingKey)          | невалидный → 401  |
| 6   | Проверить exp                                              | истёк → 401       |
| 7   | Извлечь claims: sub (userId), email, displayName, roles   | —                 |
| 8   | Создать `UsernamePasswordAuthenticationToken` с claims     | —                 |
| 9   | Поместить в `SecurityContextHolder`                        | —                 |
| 10  | `filterChain.doFilter(request, response)`                  | —                 |

### Исключения из фильтра

```java
@Override
protected boolean shouldNotFilter(HttpServletRequest request) {
    String path = request.getServletPath();
    return path.equals("/auth/login")
        || path.equals("/auth/register")
        || path.equals("/auth/refresh");
}
```

| Путь                  | Причина                |
| --------------------- | ---------------------- |
| `POST /auth/login`    | логин без токена       |
| `POST /auth/register` | регистрация без токена |
| `POST /auth/refresh`  | обновление токена      |

Все остальные пути — защищённые. При добавлении нового контроллера защита применяется **автоматически**.

### SecurityContext и AuthUser

После валидации JWT фильтр создаёт `AuthUser` и помещает его как principal в `SecurityContextHolder`:

```java
@Getter
@RequiredArgsConstructor
public class AuthUser {
    private final Long id;
    private final String email;
    private final String displayName;
    private final String role;
}
```

```java
// В JwtAuthFilter:
AuthUser user = new AuthUser(userId, email, displayName, role);
var authToken = new UsernamePasswordAuthenticationToken(
    user,                 // principal (AuthUser)
    null,                 // credentials
    List.of(new SimpleGrantedAuthority("ROLE_" + role))
);
SecurityContextHolder.getContext().setAuthentication(authToken);
```

Контроллеры получают `AuthUser` через стандартную аннотацию Spring Security `@AuthenticationPrincipal`:

```java
@GetMapping("/image-types/favorites")
public List<Long> getFavoriteIds(@AuthenticationPrincipal AuthUser user) {
    return service.getFavoriteIds(user.getId());
}
```

Из любого слоя приложения:

```java
var auth = SecurityContextHolder.getContext().getAuthentication();
AuthUser user = (AuthUser) auth.getPrincipal();
```

## @AuthenticationPrincipal AuthUser

### Механизм

`@AuthenticationPrincipal` — стандартная аннотация Spring Security. Автоматически извлекает principal из `SecurityContext` и передаёт в параметр контроллера. Кастомные аннотации и `HandlerMethodArgumentResolver` **не нужны**.

```
JwtAuthFilter
    → parse JWT → extract claims
    → new AuthUser(id, email, displayName, role)
    → SecurityContext.setAuthentication(token with AuthUser as principal)

Controller method(@AuthenticationPrincipal AuthUser user)
    → Spring Security автоматически:
       SecurityContext.getAuthentication().getPrincipal()
    → return AuthUser
```

### Поля AuthUser

| Поле          | Тип    | JWT Claim     | Пример значения      | Назначение           |
| ------------- | ------ | ------------- | -------------------- | -------------------- |
| `id`          | Long   | `sub`         | `42`                 | ID пользователя (PK) |
| `email`       | String | `email`       | `"user@example.com"` | Email                |
| `displayName` | String | `displayName` | `"Иван"`             | Отображаемое имя     |
| `role`        | String | `roles`       | `"USER"`             | Роль (USER / ADMIN)  |

### Примеры использования

```java
// Минимальный — только userId (большинство эндпоинтов)
@DeleteMapping("/{id}")
public void delete(@PathVariable Long id, @AuthenticationPrincipal AuthUser user) {
    service.delete(id, user.getId());
}

// Полный — все данные (профиль, админ-панель)
@GetMapping("/profile")
public ProfileDto profile(@AuthenticationPrincipal AuthUser user) {
    return new ProfileDto(user.getId(), user.getEmail(),
                          user.getDisplayName(), user.getRole());
}

// Проверка роли через Spring Security
@PreAuthorize("hasRole('ADMIN')")
@DeleteMapping("/admin/{id}")
public void adminDelete(@PathVariable Long id) {
    service.forceDelete(id);
}
```

## Схема прохождения запроса

### Защищённый бизнес-запрос

```
Клиент
  │  GET /image-types
  │  Authorization: Bearer <accessToken>
  ▼
┌─────────────────────────────────────────────────────────┐
│  Spring Security Filter Chain                            │
│                                                          │
│  JwtAuthFilter (OncePerRequestFilter)                    │
│  1. Путь /image-types — не в shouldNotFilter → проверяем │
│  2. Извлечь Bearer-токен из заголовка                    │
│  3. JJWT: verifyWith(signingKey) → parse claims          │
│  4. Claims → UsernamePasswordAuthenticationToken          │
│  5. SecurityContextHolder.setAuthentication(token)        │
│  6. filterChain.doFilter()                               │
├──────────────────────────────────────────────────────────┤
│  Controller                                               │
│                                                          │
│  @GetMapping("/image-types")                             │
│  public List<ImageTypeDTO> getAll()                      │
│                                                          │
│  @GetMapping("/image-types/favorites")                    │
│  public List<Long> getFavoriteIds(                        │
│      @AuthenticationPrincipal AuthUser user)               │
│  → user.getId() из SecurityContext                         │
└──────────────────────────────────────────────────────────┘
```

### Auth-запрос (проксируемый)

```
Клиент
  │  POST /auth/login
  │  { email, password }
  ▼
┌──────────────────────────────┐
│  Spring Security Filter Chain │
│  shouldNotFilter → SKIP       │
├──────────────────────────────┤
│  AuthProxyController          │
│  проксирует на auth-сервер    │
│  POST {auth-url}/login        │
│  → ответ клиенту              │
└──────────────────────────────┘
```

## Публичные и защищённые эндпоинты

### Публичные пути (без Bearer-токена)

Настраиваются в `SecurityFilterChain`:

```java
.requestMatchers("/auth/login", "/auth/register", "/auth/refresh").permitAll()
```

| Путь                  | Причина                |
| --------------------- | ---------------------- |
| `POST /auth/login`    | логин без токена       |
| `POST /auth/register` | регистрация без токена |
| `POST /auth/refresh`  | обновление токена      |

### Все остальные пути — защищённые

```java
.anyRequest().authenticated()
```

При добавлении нового контроллера с новыми путями защита применяется **автоматически** — ничего настраивать не нужно.

## Прокси-контроллер `AuthProxyController`

### Расположение

`AuthProxyController` находится в основном бэкенде (не на auth-сервере). Принимает auth-запросы от клиента и проксирует на auth-сервер.

### Маппинг URL

```
Клиент                    Основной бэкенд            Auth-сервер
POST /auth/register    →  AuthProxyController      →  POST /register
POST /auth/login       →  AuthProxyController      →  POST /login
POST /auth/refresh     →  AuthProxyController      →  POST /refresh
POST /auth/logout      →  AuthProxyController      →  POST /logout
POST /auth/change-pwd  →  AuthProxyController      →  POST /change-password
```

Клиент обращается к `/auth/*` на основном бэкенде. Auth-сервер принимает запросы **без** префикса `/auth/`.

### Защищённые auth-эндпоинты

`/auth/logout` и `/auth/change-password` проходят через `JwtAuthFilter` (не в списке исключений). Для `change-password` бэкенд извлекает `userId` из JWT и добавляет его в тело перед проксированием:

```
Клиент:    { currentPassword, newPassword }           + Bearer token
Бэкенд:   { userId, currentPassword, newPassword }   (userId из JWT)
Auth-сер:  проверяет currentPassword, обновляет hash
```

## Локальная валидация JWT

### Почему не HTTP-вызов к auth-серверу

Основной бэкенд знает signing key → может валидировать JWT самостоятельно через JJWT. Это мгновенно, без HTTP-вызовов, без кэша.

| Факт                     | Детали                                                |
| ------------------------ | ----------------------------------------------------- |
| Валидация подписи        | локально (JJWT + signing key)                         |
| Валидация exp            | локально (JJWT проверяет автоматически)               |
| HTTP к auth-серверу      | не нужен для валидации                                |
| Blacklist (jti)          | не проверяется на стороне основного бэкенда           |
| Logout-окно              | до 60 мин (TTL accessToken)                           |

### Logout-окно

После logout auth-сервер добавляет jti в blacklist. Основной бэкенд не знает о blacklist — он валидирует JWT только по подписи и exp. Токен остаётся «валидным» для основного бэкенда до истечения exp.

| Факт                 | Детали                                        |
| -------------------- | --------------------------------------------- |
| Максимальное окно    | 60 минут (TTL accessToken)                    |
| Среднее окно         | ~30 минут                                     |
| Приемлемо для MVP    | да — десктоп-приложение с доверенным клиентом  |

## Ошибки auth-модуля

| Ситуация                  | HTTP | Тело                                         |
| ------------------------- | ---- | -------------------------------------------- |
| Нет заголовка Bearer      | 401  | `{"error": "Отсутствует токен авторизации"}` |
| Невалидный/истёкший токен | 401  | `{"error": "Невалидный токен"}`              |
| Auth-сервер недоступен    | 503  | `{"error": "Сервер авторизации недоступен"}` |
| Неверный email/пароль     | 401  | `{"error": "Неверный email или пароль"}`     |
| Email уже занят           | 409  | `{"error": "Email уже зарегистрирован"}`     |

Формат ошибки: `{ "error": "текст" }` — единый для всего проекта.

```xml

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>

<!-- JJWT — локальная валидация JWT -->

    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>
    <version>0.12.6</version>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>
    <version>0.12.6</version>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-jackson</artifactId>
    <version>0.12.6</version>
    <scope>runtime</scope>
</dependency>
```

### Шаг 2. Конфигурация (application.properties)

```properties
# Auth
auth.enabled=true
auth.server.url=http://localhost:9090
jwt.signing-key=${JWT_SIGNING_KEY:default-dev-secret-key-change-in-production-must-be-at-least-32-bytes}
```

### Шаг 3. Использовать `@AuthenticationPrincipal AuthUser` в контроллерах

```java
@PostMapping
public ImageTypeDto create(@RequestBody CreateDto dto,
                           @AuthenticationPrincipal AuthUser user) {
    return service.create(dto, user.getId());
}
```

Сервисы и репозитории **не меняются** — они по-прежнему получают `Long userId`.

## Добавление нового модуля — чеклист

| #   | Действие                                                     | Нужно делать?                                            |
| --- | ------------------------------------------------------------ | -------------------------------------------------------- |
| 1   | Добавить зависимости auth                                    | нет — уже в проекте                                      |
| 2   | Настроить SecurityFilterChain                                | нет — уже зарегистрирован, работает на все пути          |
| 3   | Добавить публичный путь в исключения                         | **только если** эндпоинт должен быть доступен без токена |
| 4   | Использовать `@AuthenticationPrincipal AuthUser` вместо `@RequestHeader("UserId")` | **да** — в каждом контроллере, где нужен userId          |
| 5   | Использовать `@PreAuthorize` для проверки ролей                                     | **по необходимости**                                    |
| 6   | Изменять сервисы / репозитории                               | нет — они получают `Long userId` как и раньше            |

**Главный принцип:** Spring Security защищает все пути по умолчанию (`anyRequest().authenticated()`). Новый модуль получает авторизацию автоматически.

## Взаимодействие с таблицей `auth_users`

### Общая БД

Таблица `auth_users` находится в той же PostgreSQL, что и бизнес-таблицы.

| Действие                       | Как                                                      |
| ------------------------------ | -------------------------------------------------------- |
| Ссылаться на пользователя      | `*_user_id BIGINT REFERENCES auth_users(id)`             |
| Каскадное удаление             | `ON DELETE SET NULL` или `ON DELETE CASCADE`             |
| Получить userId                | из JWT claims через `@AuthenticationPrincipal AuthUser` (не из БД) |
| Получить email / displayName   | из `AuthUser.getEmail()` / `AuthUser.getDisplayName()`              |
| Прочитать данные из auth_users | напрямую SELECT (если нужны свежие данные, не из claims) |
| Изменить auth_users            | **нельзя** — только auth-сервер управляет этой таблицей  |
