# 05. Интеграция с основным бэкендом

## Назначение

Этот документ описывает, как **любой** модуль основного бэкенда подключает авторизацию. Auth-модуль не привязан к конкретной бизнес-логике — его механизмы (фильтр, аннотации, прокси-контроллер, кэш) универсальны и применимы к любому Spring Boot приложению в проекте.

---

## Что добавляется в основной бэкенд

Auth-модуль внедряет в существующий Spring Boot проект **четыре компонента**:

```
Основной бэкенд (любой Spring Boot проект)
│
├── JwtAuthFilter              ← OncePerRequestFilter (валидация Bearer-токена)
├── AuthController             ← Прокси auth-запросов на auth-сервер
├── @Auth* аннотации           ← Извлечение данных из JWT claims
│   ├── @AuthUserId
│   ├── @AuthEmail
│   ├── @AuthDisplayName
│   └── @AuthRole
├── SecurityContext             ← Хранилище claims текущего запроса
├── Caffeine Cache             ← Кэш ответов /validate (TTL 10 мин)
└── OkHttp клиент              ← HTTP к auth-серверу
```

Бизнес-контроллеры, сервисы и репозитории **не меняются** — они получают userId и другие данные через аннотации вместо прямого чтения заголовка.

## Схема прохождения запроса

### Защищённый бизнес-запрос (любой модуль)

```
Клиент
  │  GET /любой-эндпоинт
  │  Authorization: Bearer <accessToken>
  ▼
┌─────────────────────────────────────────────────────────┐
│  JwtAuthFilter (OncePerRequestFilter)                    │
│                                                          │
│  1. Путь в списке исключений? → пропустить               │
│  2. Извлечь Bearer-токен из заголовка                    │
│  3. SHA-256(токен) → проверить Caffeine-кэш              │
│     ├── hit (valid) → шаг 5                              │
│     └── miss → POST /validate к auth-серверу              │
│               ├── { valid: true } → кэшировать, шаг 5   │
│               └── { valid: false } → 401, стоп           │
│  4. Auth-сервер недоступен → 503, стоп                   │
│  5. Base64-decode payload → извлечь claims                │
│  6. Claims → SecurityContext (RequestAttributes)          │
│     { userId, email, displayName, roles }                │
│  7. filterChain.doFilter()                               │
├──────────────────────────────────────────────────────────┤
│  Controller (любой модуль)                                │
│                                                          │
│  @GetMapping("/любой-эндпоинт")                          │
│  public ResponseEntity<?> handle(@AuthUserId Long userId) │
│                                                          │
│  → userId извлекается из SecurityContext                  │
│  → бизнес-логика                                         │
│  → ответ                                                 │
└──────────────────────────────────────────────────────────┘
```

### Auth-запрос (проксируемый)

```
Клиент
  │  POST /auth/login
  │  { email, password }
  ▼
┌──────────────────────────────┐
│  JwtAuthFilter               │
│  путь /auth/login → SKIP     │
├──────────────────────────────┤
│  AuthController              │
│  проксирует на auth-сервер   │
│  POST {auth-url}/login       │
│  → ответ клиенту             │
└──────────────────────────────┘
```

## Подключение к существующему модулю

### Шаг 1. Зависимости (pom.xml)

```xml
<!-- Caffeine — кэш валидации токенов -->
<dependency>
    <groupId>com.github.ben-manes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
    <version>3.x</version>
</dependency>

<!-- OkHttp — HTTP-клиент к auth-серверу -->
<dependency>
    <groupId>com.squareup.okhttp3</groupId>
    <artifactId>okhttp</artifactId>
    <version>4.12.0</version>
</dependency>
```

### Шаг 2. Конфигурация (application.properties)

```properties
# Auth-сервер
auth.server.url=http://localhost:9090
auth.server.timeout-seconds=5
auth.server.retry-count=1

# Кэш валидации
auth.cache.ttl-minutes=10
auth.cache.max-size=10000
```

### Шаг 3. Регистрация фильтра

`JwtAuthFilter` регистрируется как Spring Bean. Spring Boot автоматически подхватывает его как `OncePerRequestFilter`.

### Шаг 4. Замена `@RequestHeader("UserId")` на `@AuthUserId`

**До auth-модуля:**

```java
@PostMapping
public ImageTypeDto create(@RequestBody CreateDto dto,
                           @RequestHeader("UserId") Long userId) {
    return service.create(dto, userId);
}
```

**После auth-модуля:**

```java
@PostMapping
public ImageTypeDto create(@RequestBody CreateDto dto,
                           @AuthUserId Long userId) {
    return service.create(dto, userId);
}
```

Сервис и репозиторий **не меняются** — они по-прежнему получают `Long userId`.

## Аннотации @Auth\*

### Механизм

Каждая аннотация реализована через `HandlerMethodArgumentResolver`:

```
JwtAuthFilter
    → decode JWT payload
    → SecurityContext.setAttribute("userId", 42)
    → SecurityContext.setAttribute("email", "user@example.com")
    → SecurityContext.setAttribute("displayName", "Иван")
    → SecurityContext.setAttribute("roles", "USER")

Controller method(@AuthUserId Long userId)
    → AuthUserIdResolver.resolveArgument()
    → SecurityContext.getAttribute("userId")
    → return 42
```

### Доступные аннотации

| Аннотация          | Тип    | JWT Claim     | Пример значения      | Назначение           |
| ------------------ | ------ | ------------- | -------------------- | -------------------- |
| `@AuthUserId`      | Long   | `userId`      | `42`                 | ID пользователя (PK) |
| `@AuthEmail`       | String | `email`       | `"user@example.com"` | Email                |
| `@AuthDisplayName` | String | `displayName` | `"Иван"`             | Отображаемое имя     |
| `@AuthRole`        | String | `roles`       | `"USER"`             | Роль (USER / ADMIN)  |

### Примеры использования

```java
// Минимальный — только userId (большинство эндпоинтов)
@DeleteMapping("/{id}")
public void delete(@PathVariable Long id, @AuthUserId Long userId) {
    service.delete(id, userId);
}

// Полный — все claims (профиль, админ-панель)
@GetMapping("/profile")
public ProfileDto profile(@AuthUserId Long userId,
                          @AuthEmail String email,
                          @AuthDisplayName String displayName,
                          @AuthRole String role) {
    return new ProfileDto(userId, email, displayName, role);
}

// Проверка роли
@DeleteMapping("/admin/{id}")
public void adminDelete(@PathVariable Long id, @AuthRole String role) {
    if (!"ADMIN".equals(role)) throw new IllegalStateException("Нет доступа");
    service.forceDelete(id);
}
```

## Публичные и защищённые эндпоинты

### Список публичных путей (без Bearer-токена)

| Путь                  | Причина                |
| --------------------- | ---------------------- |
| `POST /auth/login`    | логин без токена       |
| `POST /auth/register` | регистрация без токена |
| `POST /auth/refresh`  | обновление токена      |

Настраивается в `JwtAuthFilter` через метод `shouldNotFilter()`:

```java
@Override
protected boolean shouldNotFilter(HttpServletRequest request) {
    String path = request.getServletPath();
    return path.equals("/auth/login")
        || path.equals("/auth/register")
        || path.equals("/auth/refresh");
}
```

### Все остальные пути — защищённые

Любой эндпоинт, не входящий в список исключений, **требует** `Authorization: Bearer <accessToken>`. Если токен отсутствует или невалиден — `401 Unauthorized`.

Это означает, что при добавлении нового модуля (нового контроллера с новыми путями) защита применяется **автоматически** — ничего настраивать не нужно.

## Прокси-контроллер `AuthController`

### Расположение

`AuthController` находится в основном бэкенде (не на auth-сервере). Принимает auth-запросы от клиента и проксирует на auth-сервер.

### Маппинг URL

```
Клиент                    Основной бэкенд          Auth-сервер
POST /auth/register    →  AuthController         →  POST /register
POST /auth/login       →  AuthController         →  POST /login
POST /auth/refresh     →  AuthController         →  POST /refresh
POST /auth/logout      →  AuthController         →  POST /logout
POST /auth/change-pwd  →  AuthController         →  POST /change-password
```

Клиент обращается к `/auth/*` на основном бэкенде. Auth-сервер принимает запросы **без** префикса `/auth/`.

### Защищённые auth-эндпоинты

`/auth/logout` и `/auth/change-password` проходят через `JwtAuthFilter` (не в списке исключений). Для `change-password` бэкенд извлекает `userId` из JWT и добавляет его в тело перед проксированием:

```
Клиент:    { currentPassword, newPassword }           + Bearer token
Бэкенд:   { userId, currentPassword, newPassword }   (userId из JWT)
Auth-сер:  проверяет currentPassword, обновляет hash
```

## Кэширование валидации

### Зачем

Без кэша каждый бизнес-запрос = HTTP-вызов к auth-серверу (`POST /validate`). При 100 запросах/сек — 100 HTTP-вызовов/сек к auth-серверу. Кэш снижает нагрузку до ~1 вызова/10 мин на уникальный токен.

### Параметры

| Параметр   | Значение                  |
| ---------- | ------------------------- |
| Библиотека | Caffeine                  |
| Ключ       | SHA-256 от строки токена  |
| Значение   | `boolean valid`           |
| TTL        | 10 минут                  |
| Max size   | 10 000 записей            |
| Eviction   | по TTL + LRU при overflow |

### Почему SHA-256, а не сам токен

JWT-строка ~800 символов. SHA-256 = 64 символа. Экономия памяти при большом количестве кэшированных записей.

### Logout-окно

После logout токен попадает в blacklist на auth-сервере. Но кэш основного бэкенда ещё хранит `valid = true` для этого токена — **до 10 минут** после logout запрос с этим токеном пройдёт валидацию из кэша.

| Факт                 | Детали                                        |
| -------------------- | --------------------------------------------- |
| Максимальное окно    | 10 минут (TTL кэша)                           |
| Среднее окно         | ~5 минут (случайный момент logout)            |
| Приемлемо для MVP    | да — десктоп-приложение с доверенным клиентом |
| Компенсация при сбое | admin может добавить jti в blacklist вручную  |

## Ошибки auth-модуля

Исключения, добавляемые auth-модулем в `GlobalExceptionHandler`:

| Исключение                       | HTTP | Тело                                         | Когда                           |
| -------------------------------- | ---- | -------------------------------------------- | ------------------------------- |
| `AuthenticationException`        | 401  | `{"error": "Неверный email или пароль"}`     | логин с неверными данными       |
| `TokenExpiredException`          | 401  | `{"error": "Токен истёк"}`                   | accessToken exp < now           |
| `InvalidTokenException`          | 401  | `{"error": "Невалидный токен"}`              | подпись/формат/blacklist        |
| `AuthServerUnavailableException` | 503  | `{"error": "Сервер авторизации недоступен"}` | auth-сервер не отвечает/таймаут |
| `EmailAlreadyExistsException`    | 409  | `{"error": "Email уже зарегистрирован"}`     | регистрация с занятым email     |

Формат ошибки: `{ "error": "текст" }` — тот же, что и у бизнес-ошибок.

## Добавление нового модуля — чеклист

При добавлении нового бизнес-модуля (нового контроллера, сервиса, репозитория) в основной бэкенд:

| #   | Действие                                                     | Нужно делать?                                            |
| --- | ------------------------------------------------------------ | -------------------------------------------------------- |
| 1   | Добавить зависимости auth                                    | нет — уже в проекте                                      |
| 2   | Настроить JwtAuthFilter                                      | нет — уже зарегистрирован, работает на все пути          |
| 3   | Настроить кэш / auth-сервер                                  | нет — уже настроено                                      |
| 4   | Добавить публичный путь в исключения                         | **только если** эндпоинт должен быть доступен без токена |
| 5   | Использовать `@AuthUserId` вместо `@RequestHeader("UserId")` | **да** — в каждом контроллере, где нужен userId          |
| 6   | Использовать `@AuthRole` для проверки ролей                  | **по необходимости**                                     |
| 7   | Изменять сервисы / репозитории                               | нет — они получают `Long userId` как и раньше            |

**Главный принцип:** JwtAuthFilter защищает все пути по умолчанию. Новый модуль получает авторизацию автоматически — достаточно использовать `@AuthUserId` в контроллерах.

## Взаимодействие с таблицей `auth_users`

### Общая БД (вариант B — выбран)

Таблица `auth_users` находится в той же PostgreSQL, что и бизнес-таблицы. Любой модуль может:

| Действие                       | Как                                                      |
| ------------------------------ | -------------------------------------------------------- |
| Ссылаться на пользователя      | `*_user_id BIGINT REFERENCES auth_users(id)`             |
| Каскадное удаление             | `ON DELETE SET NULL` или `ON DELETE CASCADE`             |
| Получить userId                | из JWT claims через `@AuthUserId` (не из БД)             |
| Получить email / displayName   | из JWT claims через `@AuthEmail` / `@AuthDisplayName`    |
| Прочитать данные из auth_users | напрямую SELECT (если нужны свежие данные, не из claims) |
| Изменить auth_users            | **нельзя** — только auth-сервер управляет этой таблицей  |

### Правило: auth_users — read-only для основного бэкенда

Основной бэкенд **может** читать `auth_users` (SELECT, FK), но **не должен** модифицировать её (INSERT, UPDATE, DELETE). Вся мутация — через auth-сервер (API или прямой доступ к БД для админа).

## Резюме

| Факт                         | Детали                                                   |
| ---------------------------- | -------------------------------------------------------- |
| Новый модуль защищён сразу   | JwtAuthFilter применяется ко всем путям по умолчанию     |
| Доступ к userId              | `@AuthUserId` — одна аннотация в контроллере             |
| Доступ к другим claims       | `@AuthEmail`, `@AuthDisplayName`, `@AuthRole`            |
| Сервисы не меняются          | получают `Long userId` как и до auth-модуля              |
| Публичные эндпоинты          | добавляются в `shouldNotFilter()` вручную                |
| FK на auth_users             | работают (общая БД)                                      |
| Модификация auth_users       | только через auth-сервер                                 |
| Кэш валидации                | общий для всех модулей (Caffeine, TTL 10 мин)            |
| Auth-сервер — один на проект | все модули основного бэкенда используют один auth-сервер |
