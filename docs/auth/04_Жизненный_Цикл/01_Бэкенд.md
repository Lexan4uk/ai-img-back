# Бэкенд — серверная валидация и обработка авторизации

Справочник серверной логики авторизации: какие проверки делает основной сервер, как взаимодействует с auth-сервером, как валидирует токены и обрабатывает auth-запросы.

---

## Цикл обработки запросов

### Регистрация

1. Клиент отправляет `POST /auth/register` на основной бэкенд (без токена)
2. Бэкенд проксирует запрос на auth-сервер: `POST {auth-server-url}/register`
3. Auth-сервер: тримит email, проверяет формат, UNIQUE → создаёт запись, хеширует пароль (BCrypt)
4. Auth-сервер генерирует JWT-пару, записывая в claims данные нового пользователя
5. Бэкенд проксирует ответ клиенту: `{ accessToken, refreshToken }`
6. Клиент декодирует payload accessToken (base64) → получает userId, email, displayName, roles

### Логин

1. Клиент отправляет `POST /auth/login` на основной бэкенд (без токена)
2. Бэкенд проксирует на auth-сервер: `POST {auth-server-url}/login`
3. Auth-сервер: проверяет email + пароль → генерирует JWT-пару
4. Бэкенд проксирует ответ клиенту: `{ accessToken, refreshToken }`
5. Клиент декодирует payload accessToken → получает всю информацию о пользователе

### Refresh токена

1. Клиент отправляет `POST /auth/refresh` с телом `{ refreshToken }` (без accessToken)
2. Бэкенд проксирует на auth-сервер: `POST {auth-server-url}/refresh`
3. Auth-сервер: проверяет подпись и exp refreshToken → читает **актуальные данные** пользователя из БД → выдаёт новый accessToken
4. Ответ клиенту: `{ accessToken }`
5. Клиент декодирует новый accessToken → получает **обновлённую** информацию о пользователе

### Обновление данных пользователя

При изменении данных пользователя (displayName, email, role) на auth-сервере:

1. Данные обновляются в `auth_users`
2. Текущий accessToken содержит **старые** claims — он не перевыпускается мгновенно
3. Клиент выполняет `POST /auth/refresh` → auth-сервер читает свежие данные из БД → новый accessToken содержит **актуальные** claims
4. Клиент декодирует новый accessToken → видит обновлённые данные

Таким образом, refresh — это не только продление сессии, но и **механизм получения актуальных данных пользователя**. Клиент может принудительно вызвать refresh после редактирования профиля, чтобы сразу получить свежие claims.

### Валидация токена (каждый защищённый запрос)

1. Клиент отправляет любой запрос с заголовком `Authorization: Bearer <accessToken>`
2. **JwtAuthFilter** (Spring `OncePerRequestFilter`) перехватывает запрос
3. Фильтр извлекает токен из заголовка
4. Фильтр проверяет Caffeine-кэш (ключ = SHA-256 токена):
   - cache hit (`valid = true`) → переход к шагу 7
   - cache miss → `POST {auth-server-url}/validate` с телом `{ token }`, ответ кэшируется (TTL 10 минут)
5. Auth-сервер проверяет подпись + exp + **jti не в blacklist** → возвращает: `{ valid }`
6. Если `valid = false` → ответ `401 Unauthorized`, запрос не доходит до контроллера
7. Фильтр декодирует payload токена (base64) → извлекает userId → кладёт в `SecurityContext`
8. Контроллер получает userId через `@AuthUserId`

### Смена пароля

1. Клиент отправляет `POST /auth/change-password` с заголовком `Authorization: Bearer <accessToken>` и телом `{ currentPassword, newPassword }`
2. Бэкенд проксирует на auth-сервер: `POST {auth-server-url}/change-password` с телом `{ userId, currentPassword, newPassword }` (userId из JWT)
3. Auth-сервер: проверяет currentPassword (BCrypt.matches) → хеширует newPassword → UPDATE в `auth_users`
4. Ответ: `204 No Content`
5. Текущие токены остаются валидными — перелогин не требуется

Для смены пароля **обязательно знать текущий**. Если пользователь забыл пароль — он обращается к админу за временным паролем (подробнее → §8.5), входит с ним и меняет на свой.

### Logout

1. Клиент отправляет `POST /auth/logout` на основной бэкенд с заголовком `Authorization: Bearer <accessToken>` и телом `{ refreshToken }`
2. Бэкенд проксирует на auth-сервер: `POST {auth-server-url}/logout`
3. Auth-сервер извлекает `jti` (JWT ID) из обоих токенов и добавляет в таблицу `token_blacklist` с их `expires_at`
4. Оба токена остаются у клиента, но при следующем validate/refresh auth-сервер проверяет blacklist → `401`
5. Ответ: `204 No Content`

JWT нельзя «изменить» (подпись сломается). Поэтому logout = **blacklist**: auth-сервер запоминает jti отозванных токенов и отклоняет их при валидации. Записи в blacklist живут только до `expires_at` токена — после этого токен и так мёртв по exp.

## 1. Прокси-контроллер `AuthController`

Контроллер на основном бэкенде, проксирующий auth-запросы. Клиент стучится только на основной сервер.

### Эндпоинты

| Метод | Путь                    | Тело запроса                       | Auth | Проксирует на auth-сервер |
| ----- | ----------------------- | ---------------------------------- | ---- | ------------------------- |
| POST  | `/auth/register`        | `{ email, password, displayName }` | нет  | `POST /register`          |
| POST  | `/auth/login`           | `{ email, password }`              | нет  | `POST /login`             |
| POST  | `/auth/refresh`         | `{ refreshToken }`                 | нет  | `POST /refresh`           |
| POST  | `/auth/logout`          | `{ refreshToken }`                 | да   | `POST /logout`            |
| POST  | `/auth/change-password` | `{ currentPassword, newPassword }` | да   | `POST /change-password`   |

Информация о пользователе — внутри accessToken (base64-decode payload). Отдельный эндпоинт `/auth/me` не нужен.

## 2. JWT-фильтр (`JwtAuthFilter`)

### Логика фильтра

| Шаг | Действие                                                     | При ошибке                   |
| --- | ------------------------------------------------------------ | ---------------------------- |
| 1   | Читает заголовок `Authorization`                             | нет заголовка → 401          |
| 2   | Проверяет префикс `Bearer `                                  | нет префикса → 401           |
| 3   | Извлекает токен (substring после `Bearer `)                  | пустой → 401                 |
| 4   | SHA-256 от токена → проверка в Caffeine-кэше                 | —                            |
| 5   | Cache miss → `POST {auth-server-url}/validate` с `{ token }` | auth-сервер недоступен → 503 |
| 6   | Парсит ответ: `{ valid }`                                    | невалидный ответ → 500       |
| 7   | Cache miss + `valid = true` → кэширует (TTL 10 мин)          | —                            |
| 8   | `valid = false` → 401                                        | —                            |
| 9   | Декодирует payload токена (base64) → userId                  | —                            |
| 10  | Кладёт userId в `SecurityContext`                            | —                            |
| 11  | `filterChain.doFilter(request, response)`                    | —                            |

### Исключения из фильтра (публичные пути)

Фильтр **не применяется** к:

| Путь                  | Причина                |
| --------------------- | ---------------------- |
| `POST /auth/login`    | логин без токена       |
| `POST /auth/register` | регистрация без токена |
| `POST /auth/refresh`  | обновление токена      |

Настраивается через список `shouldNotFilter` в фильтре.

### Кэширование валидации (снижение нагрузки на auth-сервер)

Без кэша каждый HTTP-запрос клиента = HTTP-вызов `POST /validate` к auth-серверу. Ниже два подхода; **выбран вариант A**.

#### Вариант A — кэш ответа validate на основном бэкенде (выбран)

Основной бэкенд кэширует ответ `/validate` по ключу SHA-256 от токена с TTL 10 минут.

| Факт              | Детали                                                           |
| ----------------- | ---------------------------------------------------------------- |
| Где кэш           | основной бэкенд (in-memory, Caffeine)                            |
| Ключ              | SHA-256 от строки токена                                         |
| Значение          | `{ valid }` — ответ auth-сервера (boolean)                       |
| TTL               | 10 минут                                                         |
| Eviction          | по TTL; при получении 401 от auth-сервера — инвалидация          |
| Первый запрос     | cache miss → HTTP к auth-серверу → ответ кэшируется              |
| Повторные запросы | cache hit → ответ из памяти, без HTTP                            |
| Logout-окно       | до 10 минут после logout токен может проходить валидацию из кэша |

```
Клиент → JwtAuthFilter → кэш есть?
    → да (valid) → decode payload → userId → контроллер
    → нет → POST /validate → кэшировать → decode payload → userId → контроллер
```

#### Вариант B — локальная валидация + синхронизация blacklist (альтернативный)

Основной бэкенд сам проверяет подпись JWT (имеет signing key или public key) и периодически запрашивает blacklist у auth-сервера.

| Факт               | Детали                                                                   |
| ------------------ | ------------------------------------------------------------------------ |
| Signing key        | основной бэкенд знает ключ (симметричный) или public key (асимметричный) |
| Валидация подписи  | локально, без HTTP                                                       |
| Валидация exp      | локально, без HTTP                                                       |
| Blacklist          | периодический `GET /blacklist` к auth-серверу (раз в N минут)            |
| Хранение blacklist | in-memory Set на основном бэкенде                                        |
| Проверка jti       | `localBlacklist.contains(jti)` — без HTTP                                |
| HTTP-вызовы к auth | только периодический sync (не на каждый запрос)                          |
| Logout-окно        | до N минут (интервал синхронизации blacklist)                            |

```
Клиент → JwtAuthFilter → проверить подпись локально → проверить exp → jti в localBlacklist?
    → нет → userId из claims → контроллер
    → да → 401

Фоновая задача (каждые N минут): GET /blacklist → обновить localBlacklist
```

#### Сравнение вариантов

| Критерий                      | Вариант A (кэш validate)             | Вариант B (локальная валидация)         |
| ----------------------------- | ------------------------------------ | --------------------------------------- |
| HTTP к auth-серверу           | при cache miss (~раз в 10 мин/токен) | только периодический sync blacklist     |
| Сложность реализации          | низкая (один кэш)                    | средняя (ключ + sync + local blacklist) |
| Signing key на основном бэке  | не нужен                             | нужен (утечка = компрометация всех JWT) |
| Безопасность                  | ключ только на auth-сервере          | ключ на двух серверах                   |
| Logout-окно                   | ≤ 10 мин (TTL кэша)                  | ≤ интервал sync                         |
| Автономность при падении auth | работает пока кэш жив                | работает полностью (кроме sync)         |
| Масштабирование               | кэш на каждом инстансе               | blacklist на каждом инстансе            |

**Почему выбран вариант A:** проще в реализации, не требует распространения signing key за пределы auth-сервера, при этом даёт сопоставимое снижение нагрузки. Вариант B оправдан при очень высокой нагрузке или требовании автономности от auth-сервера.

## 3. Получение данных пользователя в контроллерах

Контроллеры получают данные из JWT claims через кастомные аннотации:

```java
@PostMapping
public ImageTypeDto create(@RequestBody CreateDto dto, @AuthUserId Long userId)

@GetMapping("/profile")
public ProfileDto profile(@AuthUserId Long userId, @AuthEmail String email,
                          @AuthDisplayName String displayName, @AuthRole String role)
```

### Аннотации

| Аннотация          | Тип    | Claim       | Назначение         |
| ------------------ | ------ | ----------- | ------------------ |
| `@AuthUserId`      | Long   | userId      | ID пользователя    |
| `@AuthEmail`       | String | email       | email пользователя |
| `@AuthDisplayName` | String | displayName | отображаемое имя   |
| `@AuthRole`        | String | roles       | роль пользователя  |

Каждая аннотация — свой `HandlerMethodArgumentResolver`, извлекающий соответствующий claim из `SecurityContext`. JwtAuthFilter при валидации декодирует payload токена и кладёт все claims в `SecurityContext`.

| Факт               | Детали                                                                              |
| ------------------ | ----------------------------------------------------------------------------------- |
| Источник данных    | JWT-токен (decode payload) → `SecurityContext`; auth-сервер подтверждает валидность |
| Подмена невозможна | данные определяются токеном, который выдал auth-сервер                              |
| Актуальность       | claims обновляются при refresh (до 60 мин устаревание)                              |


| Запрос           | Метод | Тело                                       | Ответ (успех)                   |
| ---------------- | ----- | ------------------------------------------ | ------------------------------- |
| Регистрация      | POST  | `{ email, password, displayName }`         | `{ accessToken, refreshToken }` |
| Логин            | POST  | `{ email, password }`                      | `{ accessToken, refreshToken }` |
| Валидация токена | POST  | `{ token }`                                | `{ valid }`                     |
| Refresh          | POST  | `{ refreshToken }`                         | `{ accessToken }`               |

| Смена пароля     | POST  | `{ userId, currentPassword, newPassword }` | `204 No Content`                |

### Ошибки auth-сервера

| HTTP от auth-сервера | Значение                     | Что возвращает основной бэкенд |
| -------------------- | ---------------------------- | ------------------------------ |
| 200                  | Успех                        | Проксирует ответ клиенту       |

| 500                  | Внутренняя ошибка            | 502 Bad Gateway                |
| Недоступен           | Timeout / Connection refused | 503 Service Unavailable        |

## 5. Токены

### Access Token

| Факт                | Детали                                           |
| ------------------- | ------------------------------------------------ |
| Формат              | JWT (header.payload.signature)                   |
| Время жизни         | 60 минут                                         |
| Содержит (claims)   | jti, userId, email, displayName, roles, exp, iat |
| Передаётся          | в заголовке `Authorization: Bearer <token>`      |
| Хранится на клиенте | в памяти (auth-client-utils)                     |
| Чтение claims       | base64-decode payload — не требует ключа         |

Вся информация о пользователе — внутри accessToken. Клиент декодирует payload (base64, без криптографии) и получает userId, email, displayName, roles, exp. Claim `jti` (JWT ID) — уникальный UUID, используется для blacklist при logout.

### Refresh Token

| Факт                | Детали                                                                                      |
| ------------------- | ------------------------------------------------------------------------------------------- |
| Формат              | JWT (со своим exp)                                                                          |
| Время жизни         | 30 дней                                                                                     |
| Claims              | jti, userId, exp, iat (минимальный набор — данные пользователя не нужны)                    |
| Stateless           | не хранится на сервере, валидация по подписи + exp                                          |
| Ротация             | нет — тот же refreshToken живёт до exp (30 дней) или до logout                              |
| Передаётся          | в теле `POST /auth/refresh`                                                                 |
| Хранится на клиенте | в Preferences / файле (auth-client-utils)                                                   |
| Актуализация данных | при refresh auth-сервер читает свежие данные из БД → новый accessToken с актуальными claims |

### Жизненный цикл токенов

```
login → (accessToken, refreshToken)
    → клиент декодирует accessToken → jti, userId, email, displayName, roles
    → запросы с accessToken в заголовке
    → accessToken истёк → 401
    → auth-client-utils автоматически: POST /auth/refresh
    → (новый accessToken с актуальными claims)
    → повтор оригинального запроса
    → refreshToken истёк → 401 на refresh
    → клиент перенаправляет на экран логина

logout → POST /auth/logout (accessToken + refreshToken)
    → auth-сервер добавляет jti обоих токенов в blacklist
    → следующий запрос с этими токенами → 401
    → клиент перенаправляет на экран логина
```

### Refresh как механизм актуализации данных

```
Пользователь изменил displayName на auth-сервере
    → текущий accessToken содержит старый displayName (живёт до exp)
    → клиент вызывает POST /auth/refresh
    → auth-сервер читает актуальные данные из auth_users
    → новый accessToken содержит свежий displayName
    → клиент декодирует → видит обновлённые данные
```

| Факт                   | Детали                                                       |
| ---------------------- | ------------------------------------------------------------ |
| Данные устаревают      | в пределах TTL accessToken (60 мин)                          |
| Принудительный refresh | клиент может вызвать refresh после редактирования профиля    |
| Автоматический refresh | при истечении accessToken — данные обновляются автоматически |
| Что обновляется        | все claims: email, displayName, roles                        |

## 6. Пользователи и размещение БД

Таблица `auth_users` — единственная таблица пользователей в системе. Auth-сервер управляет ей (CRUD, хеширование паролей, генерация токенов). Основной бэкенд получает userId из JWT-токена.

Ниже два варианта размещения БД; **выбран вариант B**.

#### Вариант B — общая БД (выбран)

Auth-сервер и основной бэкенд подключаются к одному PostgreSQL. Таблицы `auth_users` и `token_blacklist` лежат рядом с остальными таблицами.

| Факт                      | Детали                                                                  |
| ------------------------- | ----------------------------------------------------------------------- |
| PostgreSQL                | один инстанс, одна БД                                                   |
| `auth_users`              | в той же БД, что и `image_types`, `styles`, etc.                        |
| FK на `auth_users`        | возможны — `*_user_id → auth_users(id)`                                 |
| Каскады                   | работают — ON DELETE SET NULL / CASCADE                                 |
| Удаление пользователя     | FK каскад: `SET NULL` для `created_by_user_id`, `CASCADE` для favorites |
| userId в основном бэке    | берётся из JWT-токена (claim `userId` = `auth_users.id`)                |
| Информация о пользователе | из JWT claims (декодирование payload accessToken)                       |
| Валидация                 | по-прежнему HTTP к auth-серверу (архитектура не меняется)               |

**Преимущества:**

- FK-ограничения и каскады работают — целостность данных на уровне БД
- Таблицу `users` не нужно удалять — она переименовывается в `auth_users`
- Нет orphaned userId при удалении пользователя
- Одна БД для бэкапа и миграций
- Проще деплой (один PostgreSQL)

**Недостатки:**

- Auth-сервер и основной бэкенд связаны через общую БД (coupling на уровне схемы)
- При выделении auth в полностью независимый сервис потребуется миграция (убирать FK, выносить таблицы)

#### Вариант A — раздельные БД (альтернативный)

Auth-сервер имеет свою PostgreSQL, основной бэкенд — свою. Таблицы `auth_users` и `token_blacklist` физически в другой БД.

| Факт                  | Детали                                                  |
| --------------------- | ------------------------------------------------------- |
| PostgreSQL            | два инстанса (или два database в одном инстансе)        |
| FK на `auth_users`    | невозможны — разные БД                                  |
| `*_user_id`           | plain BIGINT без FK                                     |
| Каскады               | не работают — orphaned userId при удалении пользователя |
| Удаление пользователя | данные основного бэка остаются с неактуальным userId    |

**Преимущества:**

- Полная изоляция — auth-сервер можно вынести в отдельный кластер
- Независимые бэкапы и масштабирование БД
- Auth-сервер переиспользуем другими проектами без зависимости

**Недостатки:**

- Нет FK → нет каскадов → orphaned userId
- Два PostgreSQL для деплоя и обслуживания
- Миграция текущей `users`: DROP TABLE + DROP FK + изменение всех `*_user_id` на plain BIGINT

#### Сравнение

| Критерий                | Вариант B (общая БД)      | Вариант A (раздельные БД)      |
| ----------------------- | ------------------------- | ------------------------------ |
| FK и каскады            | работают                  | невозможны                     |
| Целостность данных      | на уровне БД              | только на уровне приложения    |
| Orphaned userId         | нет (каскады)             | да (при удалении пользователя) |
| Деплой                  | один PostgreSQL           | два PostgreSQL                 |
| Coupling                | общая схема               | полная изоляция                |
| Переиспользуемость auth | нужна миграция при выносе | готов к выносу                 |

**Почему выбран вариант B:** проще деплой, FK-каскады обеспечивают целостность данных. При необходимости полной изоляции — переход на вариант A возможен позже.

## 7. Обработчик ошибок (auth)

Исключения, обрабатываемые auth-модулем:

| Исключение                       | HTTP | Тело                                         |
| -------------------------------- | ---- | -------------------------------------------- |

## 8. Таблицы БД (auth-сервер)

### 8.1 `auth_users`

| Колонка         | Тип       | NOT NULL | UNIQUE | DEFAULT  |
| --------------- | --------- | -------- | ------ | -------- |
| `id`            | BIGSERIAL | PK       | PK     | auto     |
| `email`         | TEXT      | да       | да     | —        |
| `display_name`  | TEXT      | да       | нет    | —        |
| `password_hash` | TEXT      | да       | нет    | —        |
| `role`          | TEXT      | да       | нет    | `'USER'` |

- `display_name` хранится здесь — единственная таблица пользователей в системе.
- Роли: `USER`, `ADMIN`.

#### Хранение пароля

Ниже два подхода; **выбран вариант B**.

##### Вариант B — BCrypt-хеширование (выбран)

Пароль **хешируется** (one-way). Восстановить оригинал невозможно.

| Факт        | Детали                                                   |
| ----------- | -------------------------------------------------------- |
| Библиотека  | `spring-security-crypto` → `BCryptPasswordEncoder`       |
| Алгоритм    | BCrypt (adaptive hash)                                   |
| Ключ / salt | не нужны — salt генерируется автоматически внутри BCrypt |
| Обратимость | нет — one-way, восстановить оригинал невозможно          |

```java
BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();

// Регистрация
String hash = encoder.encode(rawPassword);  // → сохранить в БД

// Логин
boolean matches = encoder.matches(rawPassword, hash);  // → true/false
```

##### Вариант A — AES-шифрование (альтернативный)

Пароль **шифруется** (reversible encryption). Ключ и salt хранятся в переменных окружения.

| Факт        | Детали                                                       |
| ----------- | ------------------------------------------------------------ |
| Библиотека  | `spring-security-crypto` → `Encryptors.text(key, salt)`      |
| Алгоритм    | AES-256-CBC                                                  |
| Ключ / salt | переменные окружения `AUTH_ENCRYPT_KEY`, `AUTH_ENCRYPT_SALT` |
| Обратимость | да — при наличии ключа и salt можно расшифровать обратно     |

##### Сравнение

| Критерий                | Вариант B (BCrypt)               | Вариант A (AES)                         |
| ----------------------- | -------------------------------- | --------------------------------------- |
| Обратимость             | нет — one-way                    | да — расшифровка при наличии ключа      |
| При утечке БД           | пароли не восстановимы           | пароли защищены (нужен ещё ключ из env) |
| При утечке БД + ключа   | не применимо                     | все пароли раскрыты                     |
| Env-зависимость         | нет                              | да — ключ и salt в переменных окружения |
| Забытый пароль          | админ сбрасывает через БД (§8.5) | админ может расшифровать оригинал       |
| Индустриальный стандарт | да                               | нет                                     |
| Сложность реализации    | низкая (без конфигурации)        | низкая (2 env-переменные)               |

**Почему выбран вариант B:** индустриальный стандарт, нет зависимости от env-переменных, при утечке БД пароли не восстановимы. Забытый пароль решается ручным сбросом через БД (§8.5).

### 8.2 `token_blacklist`

| Колонка      | Тип         | NOT NULL | UNIQUE | DEFAULT |
| ------------ | ----------- | -------- | ------ | ------- |
| `jti`        | TEXT        | PK       | PK     | —       |
| `expires_at` | TIMESTAMPTZ | да       | нет    | —       |

- `jti` — JWT ID (UUID) из claim отозванного токена.
- `expires_at` — копия `exp` из токена. После этого момента запись бесполезна (токен и так мёртв).
- При logout: добавляются jti обоих токенов (access + refresh).
- При validate/refresh: auth-сервер проверяет `WHERE jti = :jti` — если найден, токен отклонён.

### 8.3 Очистка blacklist

Записи в `token_blacklist` нужны только до `expires_at` — после этого токен невалиден по `exp` и blacklist-запись бесполезна.

**Механизм:** Spring `@Scheduled` задача на auth-сервере:

```java
@Scheduled(fixedRate = 1, timeUnit = TimeUnit.HOURS)
public void cleanExpiredBlacklistEntries() {
    blacklistRepository.deleteByExpiresAtBefore(Instant.now());
}
```

| Факт                   | Детали                                                           |
| ---------------------- | ---------------------------------------------------------------- |
| Частота очистки        | раз в день                                                       |
| Условие удаления       | `DELETE FROM token_blacklist WHERE expires_at < NOW()`           |
| Максимальный размер    | ≈ кол-во logout-ов за TTL refreshToken (30 дней)                 |
| Типичный размер        | десятки–сотни записей (одна запись = ~80 байт: UUID + timestamp) |
| Индекс на `expires_at` | рекомендуется для эффективности DELETE                           |

При accessToken TTL = 60 мин и refreshToken TTL = 30 дней: один logout = 2 записи. Запись accessToken живёт ≤ 60 мин, запись refreshToken живёт ≤ 30 дней. При 100 logout-ах в день: ~200 записей в пике, из которых большинство очищаются в течение часа.

### 8.4 Хранение выданных токенов — нет

Токены **stateless**: JWT содержит все claims (jti, userId, email, displayName, roles, exp). Auth-сервер **не хранит** выданные токены. Валидация = проверка подписи + exp + отсутствие jti в blacklist. Refresh token — тоже JWT со своим exp. При refresh auth-сервер читает актуальные данные из `auth_users` и записывает свежие claims в новый accessToken.

### 8.5 Смена и сброс пароля

BCrypt — one-way: узнать текущий пароль из хеша невозможно.

#### Смена пароля (пользователь знает текущий пароль)

Пользователь меняет пароль через приложение: `POST /auth/change-password`.

| Факт                  | Детали                                                               |
| --------------------- | -------------------------------------------------------------------- |
| Эндпоинт              | `POST /auth/change-password` (проксируется на auth-сервер)           |
| Тело запроса          | `{ currentPassword, newPassword }`                                   |
| Auth                  | да — accessToken в заголовке, userId извлекается из JWT              |
| Проверка auth-сервера | `BCrypt.matches(currentPassword, hash)` — если не совпал → 401       |
| Валидация newPassword | минимум 6 символов (аналогично регистрации)                          |
| Действие              | `BCrypt.encode(newPassword)` → UPDATE `password_hash` в `auth_users` |
| Ответ                 | `204 No Content`                                                     |
| Токены                | остаются валидными — перелогин не требуется                          |

#### Сброс пароля админом (пользователь забыл пароль)

Эндпоинта для сброса нет. Админ выполняет сброс вручную через БД.

**Процедура:**

1. Пользователь сообщает админу (лично / мессенджер): «забыл пароль»
2. Админ генерирует временный пароль
3. Админ хеширует его через BCrypt (утилита / скрипт):

```java
new BCryptPasswordEncoder().encode("временныйПароль123")
// → "$2a$10$xK8f..."
```

4. Админ обновляет запись в БД:

```sql
UPDATE auth_users SET password_hash = '$2a$10$xK8f...' WHERE email = 'user@example.com';
```

5. Админ сообщает пользователю временный пароль (лично / мессенджер)
6. Пользователь входит с временным паролем → меняет на удобный через `POST /auth/change-password`

| Факт                        | Детали                                                         |
| --------------------------- | -------------------------------------------------------------- |
| Автоматизация сброса        | нет — ручной процесс через прямой доступ к БД                  |
| Кто выполняет сброс         | админ с доступом к PostgreSQL                                  |
| Инвалидация текущих токенов | нет — старые токены живут до exp (или ручной blacklist по jti) |

#### Полный сценарий «забыл пароль»

```
Пользователь: «забыл пароль» → админ
    → админ: BCrypt.encode("tempPass") → UPDATE в БД
    → админ → пользователю: «твой временный пароль: tempPass»
    → пользователь: POST /auth/login { email, "tempPass" } → успех
    → пользователь: POST /auth/change-password { "tempPass", "мойНовыйПароль" } → 204
    → готово — пользователь работает с новым паролем
```

## 9. Сводка пробелов валидации (auth)

| Пробел                           | Где               | Последствие                                   | Важность |
| -------------------------------- | ----------------- | --------------------------------------------- | -------- |
| Формат email                     | auth-сервер       | невалидный email принимается                  | средняя  |
| Сложность пароля                 | auth-сервер       | слабые пароли принимаются                     | средняя  |
| Rate limiting на логин           | auth-контроллер   | brute-force атаки                             | высокая  |
| Rate limiting на refresh         | auth-контроллер   | злоупотребление refresh-ами                   | средняя  |
| Refresh token в Preferences      | auth-client-utils | хранение на диске (desktop-приложение — ОК)   | низкая   |
| Принудительный отзыв (не logout) | stateless JWT     | admin должен вручную добавить jti в blacklist | низкая   |
| Устаревшие claims                | accessToken       | данные пользователя устаревают до 60 мин      | низкая   |
| Logout-окно кэша                 | JwtAuthFilter     | до 10 мин после logout токен проходит из кэша | низкая   |

**Пояснения:**

- **Rate limiting** — не реализован в MVP; при публичном деплое обязателен (минимум по IP + по email).
- **Orphaned userId** — при варианте B (общая БД) — решено FK-каскадами. При варианте A (раздельные БД) — колонки `*_user_id` сохраняют старый userId после удаления пользователя.
- **Принудительный отзыв** — logout покрывает штатный сценарий (blacklist). Для экстренного отзыва скомпрометированного токена admin добавляет jti в blacklist вручную. В MVP — допустимо.
- **Устаревшие claims** — после изменения профиля текущий accessToken содержит старые данные. Компенсация: принудительный refresh после редактирования, или автоматическое обновление при истечении TTL.
- **Logout-окно кэша** — при logout токен добавляется в blacklist на auth-сервере, но кэш на основном бэкенде продолжает возвращать `valid = true` до истечения TTL кэша (10 мин). Приемлемо для MVP.
