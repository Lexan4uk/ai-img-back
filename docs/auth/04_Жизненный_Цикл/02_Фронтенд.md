# Фронтенд — авторизация на клиенте

Справочник клиентской логики авторизации: какие проверки клиент делает локально, как управляет токенами, как обрабатывает auth-ошибки.

---

## Цикл работы пользователя (auth)

### Запуск приложения

1. Клиент проверяет наличие refreshToken в Preferences
2. Если refreshToken есть → `POST /auth/refresh`
   - Успех → декодируем accessToken, переходим на Галерею
   - 401 → refreshToken истёк, переходим на экран Логина
3. Если refreshToken нет → переходим на экран Логина

### Логин

1. Пользователь заполняет email и пароль
2. Клиентская валидация (§1) — кнопка «Войти» неактивна, пока поля невалидны
3. Клик «Войти» → `POST /auth/login` с `{ email, password }`
4. Успех → `{ accessToken, refreshToken }`
   - accessToken сохраняется в памяти
   - refreshToken сохраняется в Preferences
   - Клиент декодирует payload accessToken → userId, email, displayName, roles
   - Переход на Галерею
5. Ошибка → сообщение пользователю (§5)

### Регистрация

1. Пользователь заполняет email, пароль, подтверждение пароля, displayName
2. Клиентская валидация (§2) — кнопка «Зарегистрироваться» неактивна, пока поля невалидны
3. Клик «Зарегистрироваться» → `POST /auth/register` с `{ email, password, displayName }` (confirmPassword на сервер **не отправляется**)
4. Успех → `{ accessToken, refreshToken }`
   - Сохранение и декодирование — аналогично логину
   - Переход на Галерею
5. Ошибка → сообщение пользователю (§5)

### Работа с приложением (авторизован)

Каждый HTTP-запрос отправляется с заголовком `Authorization: Bearer <accessToken>` (подставляется OkHttp Interceptor в auth-client-utils).

При получении 401 на любой запрос:

1. auth-client-utils автоматически: `POST /auth/refresh` с текущим refreshToken
2. Успех → новый accessToken в памяти → **повтор оригинального запроса**
3. 401 на refresh → refreshToken мёртв → переход на экран Логина

### Смена пароля

1. Пользователь открывает форму смены пароля (настройки / профиль)
2. Заполняет: текущий пароль, новый пароль, подтверждение нового пароля
3. Клиентская валидация (§5) — кнопка «Сменить пароль» неактивна, пока поля невалидны
4. Клик «Сменить пароль» → `POST /auth/change-password` с `{ currentPassword, newPassword }` (confirmPassword на сервер **не отправляется**)
5. Успех (204) → toast «Пароль изменён», форма очищается
6. 401 → «Неверный текущий пароль» — показать под полем текущего пароля
7. Ошибка → сообщение пользователю (§6)

Если пользователь забыл текущий пароль — он обращается к админу за временным (подробнее → бэкенд §8.5), входит с ним и меняет на свой через эту форму.

### Logout

1. Клик «Выйти» → `POST /auth/logout` с accessToken (заголовок) и `{ refreshToken }` (тело)
2. Успех (204) или ошибка — в обоих случаях:
   - accessToken удаляется из памяти
   - refreshToken удаляется из Preferences
   - Переход на экран Логина

Logout — best-effort: даже если запрос к серверу не прошёл (сеть, 503), клиент очищает токены локально. Серверный blacklist — дополнительная защита, не блокирующая клиентский logout.

## 1. Форма логина

### `email`

| Факт             | Детали                                                        |
| ---------------- | ------------------------------------------------------------- |
| Лимит клиента    | 254 символа. Сверх — поле не принимает ввод                   |
| Проверка клиента | не пустое + формат email (regex `^[^\s@]+@[^\s@]+\.[^\s@]+$`) |
| Trim             | клиент тримит перед отправкой                                 |
| Сервер тримит    | auth-сервер тримит повторно                                   |

### `password`

| Факт             | Детали                                       |
| ---------------- | -------------------------------------------- |
| Лимит клиента    | 128 символов. Сверх — поле не принимает ввод |
| Проверка клиента | не пустое, минимум 6 символов                |
| Без trim         | пароль передаётся как есть (пробелы значимы) |

### Состояние кнопки «Войти»

Кнопка **неактивна**, пока:

- email пустой или невалидный формат
- password пустой или < 6 символов

## 2. Форма регистрации

### `email`

| Факт             | Детали                                                        |
| ---------------- | ------------------------------------------------------------- |
| Лимит клиента    | 254 символа. Сверх — поле не принимает ввод                   |
| Проверка клиента | не пустое + формат email (regex `^[^\s@]+@[^\s@]+\.[^\s@]+$`) |
| Trim             | клиент тримит перед отправкой                                 |
| 409 от сервера   | «Email уже зарегистрирован» — показать под полем email        |

### `password`

| Факт             | Детали                                       |
| ---------------- | -------------------------------------------- |
| Лимит клиента    | 128 символов. Сверх — поле не принимает ввод |
| Минимум          | 6 символов                                   |
| Проверка клиента | не пустое, ≥ 6 символов                      |
| Без trim         | пароль передаётся как есть                   |

### `confirmPassword`

| Факт             | Детали                                           |
| ---------------- | ------------------------------------------------ |
| Лимит клиента    | 128 символов (аналогично password)               |
| Проверка клиента | не пустое + совпадает с `password`               |
| Без trim         | сравнивается как есть                            |
| На сервер        | **не отправляется** — проверка только на клиенте |

### `displayName`

| Факт             | Детали                                       |
| ---------------- | -------------------------------------------- |
| Лимит клиента    | 100 символов. Сверх — поле не принимает ввод |
| Проверка клиента | не пустое (после trim)                       |
| Trim             | клиент тримит перед отправкой                |

### Состояние кнопки «Зарегистрироваться»

Кнопка **неактивна**, пока:

- email пустой или невалидный формат
- password пустой или < 6 символов
- confirmPassword пустой или не совпадает с password
- displayName пустой (после trim)

## 3. Управление токенами (auth-client-utils)

### Хранение

| Токен        | Где хранится          | Почему                                                   |
| ------------ | --------------------- | -------------------------------------------------------- |
| accessToken  | в памяти (переменная) | короткоживущий (60 мин), не нужен после перезапуска      |
| refreshToken | Preferences / файл    | долгоживущий (30 дней), нужен при перезапуске приложения |

### Декодирование accessToken

Клиент декодирует payload accessToken (base64, без криптографии) и получает:

| Claim       | Тип    | Назначение                               |
| ----------- | ------ | ---------------------------------------- |
| jti         | String | JWT ID (UUID) — клиентом не используется |
| userId      | Long   | ID пользователя                          |
| email       | String | email пользователя                       |
| displayName | String | отображаемое имя                         |
| roles       | List   | роли (USER, ADMIN)                       |
| exp         | Long   | время истечения (unix timestamp)         |
| iat         | Long   | время выдачи (unix timestamp)            |

### OkHttp Interceptor

| Факт                | Детали                                                           |
| ------------------- | ---------------------------------------------------------------- |
| Добавляет заголовок | `Authorization: Bearer <accessToken>` к каждому запросу          |
| Исключения          | `/auth/login`, `/auth/register`, `/auth/refresh` — без заголовка |
| При 401             | автоматический refresh + повтор оригинального запроса            |
| При 401 на refresh  | сигнал клиенту → переход на экран Логина                         |
| Повторная попытка   | одна (не бесконечный цикл)                                       |

### Автоматический refresh

```
Запрос → 401
    → Interceptor: POST /auth/refresh с текущим refreshToken
    → Успех → обновить accessToken в памяти → повторить оригинальный запрос
    → 401 на refresh → очистить оба токена → сигнал: «требуется логин»
```

## 4. Logout

| Факт                  | Детали                                                              |
| --------------------- | ------------------------------------------------------------------- |
| Запрос                | `POST /auth/logout` с accessToken (заголовок) + refreshToken (тело) |
| При успехе (204)      | очистка токенов, переход на Логин                                   |
| При ошибке сети / 503 | очистка токенов, переход на Логин (best-effort)                     |
| Серверный blacklist   | дополнительная защита — токен невалиден на сервере                  |
| Клиентская очистка    | основная защита — токен удалён локально                             |

## 6. Ответы на ошибки (auth)

| HTTP | Когда                        | Тело                                         | Действие клиента                             |
| ---- | ---------------------------- | -------------------------------------------- | -------------------------------------------- |
| 200  | Успех login/register/refresh | `{ accessToken, ... }`                       | сохранить токены, продолжить                 |
| 204  | Успех change-password/logout | —                                            | toast «Пароль изменён» / переход на Логин    |
| 400  | Невалидные данные            | `{"error": "текст"}`                         | показать текст ошибки                        |
| 401  | Неверный пароль / токен      | `{"error": "Неверный email или пароль"}`     | показать ошибку (логин) / auto-refresh (401) |
| 404  | Пользователь не найден       | `{"error": "текст"}`                         | показать ошибку                              |
| 409  | Email уже занят              | `{"error": "Email уже зарегистрирован"}`     | показать под полем email                     |
| 503  | Auth-сервер недоступен       | `{"error": "Сервер авторизации недоступен"}` | toast «Сервис временно недоступен»           |

### Маппинг ошибок на UI

| Ошибка сервера                                     | Где показывать            |
| -------------------------------------------------- | ------------------------- |
| «Неверный email или пароль»                        | под формой логина         |
| «Неверный текущий пароль» (401 на change-password) | под полем currentPassword |
| «Email уже зарегистрирован»                        | под полем email           |
| «Сервер авторизации недоступен»                    | toast / snackbar          |
| Сетевая ошибка (timeout, no connection)            | toast / snackbar          |
| Любая другая                                       | toast / snackbar          |

## 7. Сводка клиентских валидаций

| Поле               | Экран        | Не пустое | Min длина | Max длина | Формат        | Trim | На сервер |
| ------------------ | ------------ | --------- | --------- | --------- | ------------- | ---- | --------- |
| email              | логин, рег.  | да        | —         | 254       | email regex   | да   | да        |
| password           | логин, рег.  | да        | 6         | 128       | —             | нет  | да        |
| confirmPassword    | регистрация  | да        | 6         | 128       | = password    | нет  | нет       |
| displayName        | регистрация  | да        | —         | 100       | —             | да   | да        |
| currentPassword    | смена пароля | да        | 6         | 128       | —             | нет  | да        |
| newPassword        | смена пароля | да        | 6         | 128       | —             | нет  | да        |
| confirmNewPassword | смена пароля | да        | 6         | 128       | = newPassword | нет  | нет       |

Все проверки — **локальные**, без запроса к серверу. Кнопка отправки неактивна, пока хотя бы одна проверка не пройдена. Поля `confirmPassword` и `confirmNewPassword` на сервер не отправляются.
