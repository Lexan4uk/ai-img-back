# 02. Пользователь (auth_users)

## Назначение

Единственная таблица пользователей в системе. Хранит учётные данные (email, пароль), отображаемое имя и роль. Управляется auth-сервером (регистрация, логин, смена пароля). Основной бэкенд получает userId из JWT-токена.

---

## Структура таблицы

```sql
CREATE TABLE auth_users (
    id            BIGSERIAL  PRIMARY KEY,
    email         TEXT       NOT NULL,
    display_name  TEXT       NOT NULL,
    password_hash TEXT       NOT NULL,
    role          TEXT       NOT NULL DEFAULT 'USER',
    CONSTRAINT uq_auth_users_email UNIQUE (email)
);
```

| Поле            | Тип       | NOT NULL | UNIQUE | DEFAULT  | Описание                       |
| --------------- | --------- | -------- | ------ | -------- | ------------------------------ |
| `id`            | BIGSERIAL | PK       | PK     | auto     | Уникальный идентификатор       |
| `email`         | TEXT      | да       | да     | —        | Email (уникальный, для логина) |
| `display_name`  | TEXT      | да       | нет    | —        | Отображаемое имя               |
| `password_hash` | TEXT      | да       | нет    | —        | BCrypt-хеш пароля              |
| `role`          | TEXT      | да       | нет    | `'USER'` | Роль: `USER` или `ADMIN`       |

---

## Хранение пароля

| Факт        | Детали                                                   |
| ----------- | -------------------------------------------------------- |
| Библиотека  | `spring-security-crypto` → `BCryptPasswordEncoder`       |
| Алгоритм    | BCrypt (adaptive hash, one-way)                          |
| Ключ / salt | не нужны — salt генерируется автоматически внутри BCrypt |
| Обратимость | нет — восстановить оригинал из хеша невозможно           |

---

## API (auth-сервер)

Auth-сервер — единственный, кто пишет в `auth_users`. Основной бэкенд проксирует запросы клиента.

| Действие      | Что происходит с `auth_users`                                  |
| ------------- | -------------------------------------------------------------- |
| Регистрация   | INSERT — email (UNIQUE), display_name, BCrypt.encode(password) |
| Логин         | SELECT — проверка email + BCrypt.matches(password, hash)       |
| Смена пароля  | UPDATE password_hash — BCrypt.encode(newPassword)              |
| Сброс (админ) | UPDATE password_hash — вручную через SQL                       |

## DTO

```java
// Ответ (внутри JWT claims, не как отдельный DTO)
// userId, email, displayName, roles — декодируются из accessToken payload

// Запрос регистрации
class RegisterRequest {
    String email;
    String password;
    String displayName;
}

// Запрос логина
class LoginRequest {
    String email;
    String password;
}

// Запрос смены пароля
class ChangePasswordRequest {
    String currentPassword;
    String newPassword;
}
```

## Бизнес-правила

1. **Email уникален** — constraint на уровне БД. Повторный → 409.
2. **Email тримится** auth-сервером перед записью.
3. **Пароль хешируется BCrypt** — в БД хранится только хеш.
4. **Роль по умолчанию** — `USER`. Назначение `ADMIN` — вручную через SQL.
5. **display_name** — обязательное поле, хранится здесь (единственная таблица пользователей).
6. **Данные пользователя в JWT** — email, displayName, roles записываются в claims при выдаче токена. Обновляются при refresh.

## Связи с другими модулями

```
auth_users.id  ←  *_user_id BIGINT REFERENCES auth_users(id)
                   (любая бизнес-таблица любого модуля)
```

| Факт                     | Детали                                                                      |
| ------------------------ | --------------------------------------------------------------------------- |
| Внешние FK               | бизнес-модули могут ссылаться на `auth_users.id`                            |
| ON DELETE                | определяется бизнес-модулем (SET NULL для сохранения данных, CASCADE и др.) |
| Модификация `auth_users` | только auth-сервер — бизнес-модули не пишут в эту таблицу                   |
| Чтение `auth_users`      | бизнес-модули могут SELECT (если нужны свежие данные, не из JWT claims)     |

Auth-модуль **не знает** о конкретных бизнес-таблицах. Ответственность за FK и каскады — на стороне бизнес-модуля.
