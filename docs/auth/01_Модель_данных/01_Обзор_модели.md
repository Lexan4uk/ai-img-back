# 01. Обзор модели данных (auth)

## Общее описание

Auth-модуль хранит данные в PostgreSQL (общая БД с основным бэкендом — вариант B). Первичные ключи — `BIGSERIAL` (auto-increment), кроме `token_blacklist`, где PK — текстовый `jti`.

Данные разделены на 2 слоя:

1. **Пользователи** — учётные записи, пароли, роли
2. **Безопасность** — blacklist отозванных токенов

---

## ER-диаграмма

```
┌─────────────────────┐
│     auth_users      │
│─────────────────────│
│ id (PK, BIGSERIAL)  │
│ email (UNIQUE)      │
│ display_name        │
│ password_hash       │
│ role                │
└─────────────────────┘

┌─────────────────────┐
│  token_blacklist    │
│─────────────────────│
│ jti (PK, TEXT)      │
│ expires_at          │
└─────────────────────┘
```

Таблицы `auth_users` и `token_blacklist` не связаны между собой FK. `token_blacklist` хранит jti (UUID из JWT claim), а не userId. Бизнес-таблицы других модулей могут ссылаться на `auth_users.id` через FK — это ответственность тех модулей.

---

## Таблицы

| № | Таблица            | Назначение                                     | Записей (ожидание) |
| - | ------------------ | ---------------------------------------------- | ------------------ |
| 1 | `auth_users`       | Пользователи: email, пароль, роль              | десятки            |
| 2 | `token_blacklist`  | Отозванные токены (logout) — jti + expires_at   | десятки–сотни      |

---

## Связь с основным бэкендом (вариант B — общая БД)

Auth-таблицы лежат в одной PostgreSQL с бизнес-таблицами основного бэкенда.

| Факт                        | Детали                                                                |
| --------------------------- | --------------------------------------------------------------------- |
| FK на `auth_users.id`       | бизнес-таблицы могут ссылаться через `*_user_id BIGINT REFERENCES auth_users(id)` |
| ON DELETE                   | определяется бизнес-модулем (SET NULL, CASCADE и т.д.)               |
| Модификация `auth_users`    | только auth-сервер — основной бэкенд и другие модули читают, но не пишут |
| userId в бизнес-модулях     | берётся из JWT claims через `@AuthUserId`, не из прямого SELECT      |

---

## Ключевые инварианты

1. **`auth_users` — единственная таблица пользователей** в системе. Старая `users` переименовывается в `auth_users` при миграции.
2. **Email уникален** — constraint на уровне БД.
3. **Пароль хешируется BCrypt** — one-way, восстановить оригинал невозможно.
4. **Роли:** `USER` (default), `ADMIN`.
5. **Токены не хранятся** — JWT stateless. `token_blacklist` хранит только jti отозванных токенов.
6. **Очистка blacklist** — `@Scheduled` задача удаляет записи с `expires_at < NOW()` раз в день.
