# 03. Чёрный список токенов (token_blacklist)

## Назначение

Хранит jti (JWT ID) отозванных токенов после logout. JWT — stateless: «отменить» выданный токен невозможно, поэтому auth-сервер запоминает jti и отклоняет их при валидации. Записи живут до `expires_at` токена — после этого токен и так мёртв по exp.

---

## Структура таблицы

```sql
CREATE TABLE token_blacklist (
    jti        TEXT        PRIMARY KEY,
    expires_at TIMESTAMPTZ NOT NULL
);

CREATE INDEX idx_token_blacklist_expires_at ON token_blacklist (expires_at);
```

| Поле         | Тип         | NOT NULL | UNIQUE | DEFAULT | Описание                              |
| ------------ | ----------- | -------- | ------ | ------- | ------------------------------------- |
| `jti`        | TEXT        | PK       | PK     | —       | JWT ID (UUID) из claim токена         |
| `expires_at` | TIMESTAMPTZ | да       | нет    | —       | Копия `exp` из токена (когда истекает) |

Индекс на `expires_at` — для эффективности периодической очистки (`DELETE WHERE expires_at < NOW()`).

---

## Как попадают записи

При logout auth-сервер извлекает `jti` из refreshToken и добавляет в blacklist:

| Токен        | jti                                  | expires_at                    |
| ------------ | ------------------------------------ | ----------------------------- |
| refreshToken | UUID из claim `jti`                  | exp токена (через ≤ 30 дней)  |

Один logout = 1 запись в blacklist. accessToken не добавляется — его jti не проверяется ни на основном бэкенде (локальная валидация), ни при refresh. Он истечёт сам по exp (до 60 мин).

---

## Проверка при валидации

Auth-сервер при `POST /validate` и `POST /refresh`:

```sql
SELECT 1 FROM token_blacklist WHERE jti = :jti
```

- Найден → токен отозван → 401
- Не найден → токен валиден (если подпись и exp тоже ОК)

---

## Очистка

Записи с `expires_at < NOW()` бесполезны — токен и так невалиден по exp. Очистка — `@Scheduled` задача на auth-сервере:

```java
@Scheduled(fixedRate = 1, timeUnit = TimeUnit.DAYS)
public void cleanExpiredBlacklistEntries() {
    blacklistRepository.deleteByExpiresAtBefore(Instant.now());
}
```

| Факт                | Детали                                                           |
| ------------------- | ---------------------------------------------------------------- |
| Частота очистки     | раз в день                                                       |
| SQL                 | `DELETE FROM token_blacklist WHERE expires_at < NOW()`           |
| Максимальный размер | ≈ кол-во logout-ов за TTL refreshToken (30 дней)                 |
| Типичный размер     | десятки–сотни записей (одна запись ≈ 80 байт: UUID + timestamp)  |

---

## Бизнес-правила

1. **PK — текстовый jti**, не BIGSERIAL. JWT ID (UUID) генерируется auth-сервером при выдаче токена.
2. **Нет FK на `auth_users`** — blacklist хранит только jti, не userId. Один пользователь может иметь 0..N записей.
3. **Записи самоочищаются** — после `expires_at` запись не нужна.
4. **INSERT-only** (кроме очистки) — записи не обновляются, только добавляются при logout и удаляются при очистке.
5. **При сбросе пароля админом** — старые токены НЕ инвалидируются автоматически. Живут до своего exp.

---

## Связи

```
token_blacklist — изолированная таблица, нет FK.
```

Таблица не ссылается на `auth_users` и не имеет входящих FK.
