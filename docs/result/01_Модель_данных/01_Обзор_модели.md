# 01. Обзор модели данных

## Общее описание

Система хранит данные в PostgreSQL. Все первичные ключи — `BIGSERIAL` (auto-increment).
UUID не используется ни в ключах, ни в FK — только целочисленные идентификаторы.

Данные разделены на 3 слоя:

1. **Справочники** — пользователи, типы изображений, стили, избранное
2. **Контент** — ассеты (сгенерированные изображения)
3. **Генерация** — батчи и реквесты (история запусков AI)

## Well-known ID

| Константа            | Значение | Назначение                                           |
| -------------------- | -------- | ---------------------------------------------------- |
| `UNDEFINED_TYPE_ID`  | `1`      | «Неопределённый» тип — fallback при удалении типа    |
| `UNDEFINED_STYLE_ID` | `1`      | «Неопределённый» стиль — fallback при удалении стиля |

Seed-данные вставляются миграцией. Эти записи **неудаляемы** (бизнес-правило в коде).

## Ключевые инварианты

1. **Все PK — BIGSERIAL.** Порядок вставки = порядок ID. Используется для сортировки вместо `created_at` где возможно.
2. **Удаление пользователя** — данные остаются (`ON DELETE SET NULL`), удаляется только избранное (`CASCADE`).
3. **Удаление типа/стиля** — связанные assets и requests переназначаются на «Неопределённый» (через код, в транзакции), избранное удаляется (`CASCADE`).
4. **Таблица `prompts` отсутствует** — данные промпта хранятся в `generation_batches`.
5. **Статус batch не хранится** — вычисляется по COUNT реквестов (DONE/FAILED/SKIPPED).
6. **`created_at`** есть только в `assets` и `generation_requests` (для сортировки и хронологии).
