# 07. Генерация (generation_batches + generation_requests)

## Назначение

Две таблицы фиксируют историю генерации изображений:

- **`generation_batches`** — один запуск генерации. Хранит промпт пользователя,
  параметры генерации, провайдер AI, флаг перезаписи дубликатов.
- **`generation_requests`** — одна пара (тип × стиль) внутри batch.
  Хранит хеш для дедупликации, статус выполнения и ссылку на созданный ассет.

---

## Структура таблиц

### generation_batches

```sql
CREATE TABLE generation_batches (
    id                    BIGSERIAL PRIMARY KEY,
    owner_user_id         BIGINT    REFERENCES users(id) ON DELETE SET NULL,
    user_prompt           TEXT      NOT NULL,
    generation_params     JSONB     NOT NULL DEFAULT '{}',
    overwrite_duplicates  BOOLEAN   NOT NULL DEFAULT false,
    provider              TEXT      NOT NULL,
    model                 TEXT
);
```

| Поле                   | Тип       | Описание                                                 |
| ---------------------- | --------- | -------------------------------------------------------- |
| `id`                   | BIGSERIAL | PK                                                       |
| `owner_user_id`        | BIGINT    | Кто запустил (NULL если пользователь удалён)             |
| `user_prompt`          | TEXT      | Текст промпта от пользователя                            |
| `generation_params`    | JSONB     | Параметры генерации: `{"aspectRatio": "1:1"}`            |
| `overwrite_duplicates` | BOOLEAN   | `false` — пропускать дубликаты, `true` — пересоздать все |
| `provider`             | TEXT      | Провайдер AI (по умолчанию `openai`)                     |
| `model`                | TEXT      | Модель AI (nullable, например `gpt-image-1.5`)           |

> **Статуса нет.** Вычисляется по COUNT реквестов:
> все DONE → batch DONE, есть FAILED → есть ошибки, и т.д.

### generation_requests

```sql
CREATE TABLE generation_requests (
    id                BIGSERIAL   PRIMARY KEY,
    batch_id          BIGINT      NOT NULL REFERENCES generation_batches(id) ON DELETE CASCADE,
    image_type_id     BIGINT      NOT NULL REFERENCES image_types(id),
    style_id          BIGINT      NOT NULL REFERENCES styles(id),
    final_prompt_hash TEXT        NOT NULL,
    status            TEXT        NOT NULL DEFAULT 'PENDING',
    created_asset_id  BIGINT      REFERENCES assets(id),
    error_message     TEXT,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT ck_requests_status CHECK (status IN ('PENDING','RUNNING','DONE','FAILED'))
);
```

| Поле                | Тип         | Описание                                             |
| ------------------- | ----------- | ---------------------------------------------------- |
| `id`                | BIGSERIAL   | PK                                                   |
| `batch_id`          | BIGINT      | Ссылка на batch (CASCADE при удалении)               |
| `image_type_id`     | BIGINT      | Тип изображения для этой пары                        |
| `style_id`          | BIGINT      | Стиль для этой пары                                  |
| `final_prompt_hash` | TEXT        | SHA-256 от `finalPrompt + "\|" + generationParams`   |
| `status`            | TEXT        | Текущий статус (см. ниже)                            |
| `created_asset_id`  | BIGINT      | Ссылка на созданный ассет (если DONE)                |
| `error_message`     | TEXT        | Текст ошибки (если FAILED)                           |
| `created_at`        | TIMESTAMPTZ | Время создания реквеста                              |

---

## Жизненный цикл request

```
PENDING → RUNNING → DONE      (успех, создан asset)
                  → FAILED    (ошибка AI)
```

Дубликаты не получают отдельный статус — при `overwriteDuplicates: false` запрос для дубликата **не создаётся** в БД.

## API

### Проверка дубликатов

```
POST /generations/check
```

```json
{
	"userPrompt": "кузнец куёт меч",
	"generationParams": "{\"aspectRatio\": \"1:1\"}",
	"imageTypeIds": [2, 3],
	"styleIds": [5]
}
```

**Ответ:**

```json
{
	"totalCount": 6,
	"duplicateCount": 2,
	"newCount": 4
}
```

Header `UserId` **не нужен** — проверка не создаёт записей.

### Генерация

```
POST /generations
UserId: <Long>
```

```json
{
	"userPrompt": "кузнец куёт меч",
	"generationParams": "{\"aspectRatio\": \"1:1\"}",
	"imageTypeIds": [2, 3],
	"styleIds": [5],
	"overwriteDuplicates": false,
	"provider": "openai",
	"model": "gpt-image-1.5"
}
```

| Поле                   | Обязательно | По умолчанию |
| ---------------------- | ----------- | ------------ |
| `userPrompt`           | ✅          | —            |
| `generationParams`     | —           | `"{}"`       |
| `imageTypeIds`         | ✅          | —            |
| `styleIds`             | ✅          | —            |
| `overwriteDuplicates`  | —           | `false`      |
| `provider`             | —           | `openai`     |
| `model`                | —           | —            |

**Ответ** — массив по каждой паре (тип × стиль), для которых были созданы запросы:

```json
[
	{
		"requestId": 42,
		"imageTypeId": 2,
		"styleId": 5,
		"status": "DONE",
		"createdAssetId": 123,
		"errorMessage": null
	}
]
```

При `overwriteDuplicates: false` дубликаты не попадают в ответ (для них не создаются запросы).

## Алгоритм генерации

### Проверка дубликатов (`POST /generations/check`)

1. **Валидация**: промпт не пустой, списки не пустые, нет `UNDEFINED` ID
2. **Загрузка справочников**: типы и стили по ID (404 если не найдены)
3. **Подсчёт**: для каждой пары (тип × стиль) — собрать `finalPrompt`, посчитать hash, проверить `existsByHash`
4. **Ответ**: `{totalCount, duplicateCount, newCount}`

Проверка не создаёт записей в БД. Клиент показывает результат пользователю и спрашивает: пропустить дубликаты или пересоздать.

### Генерация (`POST /generations`)

1. **Валидация**: промпт не пустой, списки не пустые, нет `UNDEFINED` ID
2. **Загрузка справочников**: типы и стили по ID (404 если не найдены)
3. **Создание batch**: сохраняем промпт, params, overwriteDuplicates, provider, model
4. **Формирование реквестов**: декартово произведение types × styles. Для каждой пары:
   - Собрать `finalPrompt` по шаблону `ai.prompt-template`
   - Посчитать `hash` = `SHA-256(finalPrompt + "|" + generationParams)`
   - Если `overwriteDuplicates: false` и `existsByHash(hash)` → **пропуск** (request не создаётся)
   - Иначе → `INSERT generation_request`
5. **Последовательное выполнение**: для каждого реквеста:
   - Статус → RUNNING
   - Вызвать AI → сохранить файл → создать asset → статус DONE
   - При auth-ошибке (401/402/403) → FAILED + все оставшиеся тоже FAILED
   - При другой ошибке: `status = FAILED`, `error_message`
6. **Ответ**: массив результатов (только созданные запросы, без дубликатов)

## Дедупликация

### Зачем нужна

Генерация изображения через AI стоит денег (~$0.02–0.19 за картинку). Если два пользователя (или один пользователь дважды) отправляют одинаковый запрос — нет смысла платить за генерацию повторно. Дедупликация экономит бюджет и ускоряет ответ.

### Как работает хеш

Каждый запрос на генерацию получает уникальный **отпечаток** — SHA-256 хеш:

```
hash = SHA-256( finalPrompt + "|" + generationParams )
```

Где:
- `finalPrompt` — итоговый промпт, собранный из шаблона с подставленными типом, стилем и текстом пользователя
- `generationParams` — JSON с параметрами (aspect ratio и др.)

**Что это значит на практике:**

| Ситуация                                                  | Одинаковый хеш? | Результат                  |
| --------------------------------------------------------- | --------------- | -------------------------- |
| Тот же промпт, тот же тип, тот же стиль, те же параметры | Да              | Дубликат                   |
| Тот же промпт, но **другой стиль**                        | Нет             | Новая генерация            |
| Тот же промпт, но **другой aspect ratio**                 | Нет             | Новая генерация            |
| Тот же промпт, **другой пользователь**                    | Да              | Дубликат                   |

Хеш хранится в поле `final_prompt_hash` таблицы `generation_requests`.

### Двухшаговый процесс

Дедупликация реализована на стороне клиента:

1. Клиент вызывает `POST /generations/check` — сервер подсчитывает дубликаты (пары, для которых `existsByHash` = true)
2. Если дубликаты есть — клиент показывает диалог: «Всего N, дубликатов M. Пропустить / Пересоздать?»
3. Клиент вызывает `POST /generations` с `overwriteDuplicates: false` (пропустить) или `true` (пересоздать)

При `overwriteDuplicates: false` — дубликаты **не создают записей** в `generation_requests`. Они просто пропускаются на этапе формирования реквестов (`continue`).

При `overwriteDuplicates: true` — генерация выполняется для всех пар заново. Создаётся новый asset. Полезно, когда пользователь хочет получить другой вариант картинки.

### Область проверки

Дедупликация **глобальная** — проверяется по всем пользователям. Если пользователь A уже сгенерировал картинку с таким промптом, для пользователя B она будет считаться дубликатом. Это максимально экономит бюджет.

### Пример

```
Пользователь отправляет: 3 типа × 2 стиля = 6 пар.
Из них 2 уже были сгенерированы ранее.

overwriteDuplicates: false → 4 генерации (платно), 2 пропущены
overwriteDuplicates: true  → 6 генераций (все платно)
```

## Сборка промпта

Промпт формируется по настраиваемому шаблону (см. `ai.prompt-template` в application.properties):

```
Шаблон: "Create an image of type: {type}. Style: {style}. Description: {prompt}."

Подстановка:
  {type}   ← type_prompt из БД (или "любой" если пусто)
  {style}  ← style_prompt из БД (или "любой" если пусто)
  {prompt} ← текст пользователя

Результат:
  "Create an image of type: Photography. Style: Minimalism. Description: A blacksmith forging a sword."

Хеш:
  SHA-256("Create an image of type: Photography..." + "|" + '{"aspectRatio":"1:1"}')
  → "a3f8c1d2e5b7..."
```

## Узкие места и ограничения

### Последовательная генерация

Все пары тип × стиль обрабатываются **строго последовательно** в одном потоке.
Каждый запрос к AI ждёт завершения предыдущего.

| Сценарий              | Время ожидания (OpenAI)      | Время ожидания (Stability) |
| --------------------- | ---------------------------- | -------------------------- |
| 3 типа × 2 стиля (6) | 48–180 сек (≈ 1–3 мин)      | 30–60 сек                  |
| 5 × 4 (20)            | 160–600 сек (≈ 3–10 мин)    | 100–200 сек                |
| 10 × 10 (100)         | 800–3000 сек (≈ 13–50 мин)  | 500–1000 сек               |

Клиент блокируется на весь период генерации (один HTTP-запрос).
Нет ни прогресс-бара, ни возможности отмены.

---

### Размер batch

Сервер не ограничивает количество типов и стилей. Клиент лимитирует выбор: максимум 10 типов × 10 стилей = 100 запросов за одну генерацию.

---

### Индекс для дедупликации

Поиск дубликата выполняется через `existsByHash(hash)` — полнотабличный скан
по `final_prompt_hash` в таблице `generation_requests`.

При росте таблицы (>100K записей) этот запрос замедлится.

```sql
CREATE INDEX idx_gen_requests_hash ON generation_requests (final_prompt_hash);
```
