# Пользователь — отдельный блок

## Почему отдельно

Блок пользователя — внешняя зависимость для основного цикла приложения. Генерация, типы, стили, assets работают сами по себе и требуют от юзера ровно одно: **валидный `userId`** в заголовке запроса. Как именно этот userId получен (регистрация, логин, API-ключ, хардкод) — основному циклу безразлично.

Поэтому блок пользователя вынесен из описаний бэкенда и фронтенда — его можно реализовать/заменить независимо, не трогая логику генерации.

## Что приложению нужно от блока пользователя

1. **Получить `userId` (Long)** — при первом запуске или логине. Этот id передаётся в заголовке `UserId` во все запросы, требующие идентификации.
2. **Гарантировать, что пользователь существует в БД** — иначе FK-ограничения при создании типов/стилей/batch-ей вернут ошибку.

Всё остальное (аутентификация, права, профиль) — опционально и зависит от выбранного подхода.

---

## Временное решение

Полноценная аутентификация не реализована. Вместо неё — упрощённая схема, достаточная для desktop-приложения с одним пользователем на машину.

**Схема работы:**

1. Клиент при первом запуске создаёт пользователя (`POST /users` с email и displayName)
2. Сохраняет полученный `userId` локально (файл/preferences)
3. Передаёт `userId` в заголовке `UserId` в каждом запросе

**Эндпоинты:** CRUD без Update — `POST /users`, `GET /users`, `GET /users/{id}`, `DELETE /users/{id}`.

**Таблица `users`:** `id` (BIGSERIAL PK), `email` (TEXT NOT NULL UNIQUE), `display_name` (TEXT NOT NULL).

**Ограничения временного решения:**

- Нет аутентификации (пароль, сессия, JWT, OAuth) — любой может подставить любой UserId
- Нет валидации email формата — любая строка принимается
- Нет проверки существования userId при использовании заголовка — при несуществующем userId FK-ошибка → 400
- Нет обновления профиля (PUT/PATCH)

---

## Как UserId используется

| Контроллер           | Эндпоинты                   | Зачем                                           |
| -------------------- | --------------------------- | ----------------------------------------------- |
| GenerationController | `POST /generations`         | `owner_user_id` в batch                         |
| ImageTypeController  | `POST`, `DELETE`, favorites | создатель, проверка прав на удаление, избранное |
| StyleController      | `POST`, `DELETE`, favorites | аналогично                                      |

Заголовок передаётся как `@RequestHeader("UserId") Long currentUserId`.

---

## Каскады при удалении пользователя

| Таблица                | Колонка              | Правило  | Эффект                              |
| ---------------------- | -------------------- | -------- | ----------------------------------- |
| `image_types`          | `created_by_user_id` | SET NULL | тип остаётся, автор обнуляется      |
| `styles`               | `created_by_user_id` | SET NULL | стиль остаётся, автор обнуляется    |
| `generation_batches`   | `owner_user_id`      | SET NULL | batch остаётся, владелец обнуляется |
| `user_favorite_types`  | `user_id`            | CASCADE  | избранные типы удаляются            |
| `user_favorite_styles` | `user_id`            | CASCADE  | избранные стили удаляются           |

Удаление пользователя не удаляет его типы, стили, batch-и и assets — только обнуляет ссылку на автора и удаляет избранное.
