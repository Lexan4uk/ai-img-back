# Фронтенд — что бэкенд проверяет и что нет

Справочник для понимания, какие ограничения уже есть на сервере, а какие отсутствуют и могут привести к ненужным запросам или ошибкам.

---

## Цикл работы пользователя

### Запуск → Галерея

Приложение открывается на экране **Галерея**. Слева — список типов изображений, справа — сетка картинок.

1. При старте загружаются типы, стили и избранное (параллельно)
2. Пользователь выбирает тип → загружается первая страница assets (20 шт.)
3. Прокрутка к концу списка → подгружается следующая страница (§9)
4. Опционально выбирает стиль → assets фильтруются
5. Клик по карточке → lightbox с деталями
6. При наведении на карточку - возникает кнопка загрузки
7. Нет удаления и скачивания assets (подробнее → §8)

### Галерея → Генерация

Переход через боковую панель (2 вкладки: Галерея, Генерация).

### Генерация — заполнение формы

1. **Промпт** — текстовое поле, 3–5 строк. Обязателен для генерации
2. **Типы** — чекбоксы, минимум один. id=1 скрыт. Можно создавать новые (§1), отмечать избранным (§5)
3. **Стили** — чекбоксы, минимум один. id=1 скрыт. Аналогично типам (§2)
4. **Модель** — dropdown из 3 фиксированных моделей (§3, таблица моделей)
5. **Aspect ratio** — dropdown, зависит от выбранной модели. При смене модели — авто-сброс

Кнопка «Генерировать» **неактивна**, пока не заполнены промпт + хотя бы один тип + хотя бы один стиль.

### Генерация — отправка

1. Клик «Генерировать» → `POST /generations/check` (подробнее → §3, дедупликация)
2. Если есть дубликаты → диалог: «Пропустить» / «Пересоздать все»
3. Если нет → сразу `POST /generations`
4. Количество запросов к AI = типы × стили. Бэк не ограничивает (§3, количество запросов)

### Генерация — результат

Ответ — список `GenerationResult` по каждой паре (тип, стиль): статус DONE или FAILED с сообщением об ошибке.

### Создание типов и стилей

Диалог с двумя полями: имя (обязательно) и промпт (обязательно). Ограничения — §1, §2.

### Актуальность данных между экранами

`GalleryScreenModel` и `GenerationScreenModel` — factory-scoped (подробнее — [03*Клиентская*архитектура.md](../03_Архитектура/03_Клиентская_архитектура.md), §ScreenModel-ы). Каждый экземпляр хранит собственную копию списков типов и стилей в `StateFlow`. Общего состояния между экранами нет.

- Создание/удаление типа или стиля на экране «Генерация» → изменение видно сразу (StateFlow текущего ScreenModel обновляется).
- Экран «Галерея» увидит изменение только при переходе на вкладку — Voyager создаёт новый factory-экземпляр ScreenModel, который загружает актуальные данные из репозитория.

## 3. Генерация

### `userPrompt`

| Факт             | Детали                                         |
| ---------------- | ---------------------------------------------- |
| Лимит клиента    | 2 000 символов. Сверх — поле не принимает ввод |
| Проверка клиента | null/blank — локально, кнопка неактивна        |
| Сервис тримит    | перед сохранением                              |

### `imageTypeIds` и `styleIds`

| Факт                     | Детали                                                                   |
| ------------------------ | ------------------------------------------------------------------------ |
| Пустой список невозможен | кнопка генерации неактивна, пока не выбран хотя бы один тип и один стиль |
| Лимит клиента            | по 10 каждого. Сверх — чекбоксы заблокированы, подсказка «максимум 10»   |

### Количество запросов

- Каждая комбинация `(imageTypeId, styleId)` = 1 запрос к AI-провайдеру.
- Лимит 10 × 10 — максимум 100 запросов за одну генерацию.
- Без лимита пользователь может отправить сотни запросов одним кликом → расход бюджета API, долгое ожидание, нагрузка на сервер. 100 — достаточно для покрытия реальных сценариев, но не позволяет случайно сгенерировать тысячи картинок.

### Выбор модели (provider, model, aspectRatio)

| Факт                 | Детали                                                               |
| -------------------- | -------------------------------------------------------------------- |
| Фиксированный список | 3 модели в `AiModelRegistry`, выбор через dropdown                   |
| Провайдер            | определяется выбранной моделью, всегда `"openai"` или `"stability"`  |
| Модель               | определяется выбранной моделью                                       |
| Aspect ratio         | dropdown с допустимыми значениями для текущей модели                 |
| При смене модели     | если текущий ratio не поддерживается — сбрасывается на дефолт модели |

**Модели:**

| Отображаемое имя            | Провайдер / модель        | Aspect ratios                                   |
| --------------------------- | ------------------------- | ----------------------------------------------- |
| OpenAI GPT Image 1 Mini     | openai / gpt-image-1-mini | 1:1, 1:1.5, 1.5:1, auto                         |
| OpenAI GPT Image 1.5        | openai / gpt-image-1.5    | 1:1, 1:1.5, 1.5:1, auto                         |
| Stability Stable Image Core | stability / core          | 1:1, 16:9, 9:16, 3:2, 2:3, 4:5, 5:4, 21:9, 9:21 |

### `generationParams`

| Факт                 | Детали                                            |
| -------------------- | ------------------------------------------------- |
| Формируется клиентом | JSON `{"aspectRatio": "..."}` из выбранного ratio |

### Дедупликация (overwriteDuplicates)

| Факт                         | Детали                                                       |
| ---------------------------- | ------------------------------------------------------------ |
| Двухшаговый процесс          | сначала `POST /generations/check`, потом `POST /generations` |
| check возвращает             | `{ totalCount, duplicateCount, newCount }`                   |
| Если дубликаты есть          | диалог: «Пропустить дубликаты» / «Пересоздать все»           |
| Если дубликатов нет          | генерация сразу, без диалога                                 |
| `overwriteDuplicates: false` | сервер не создаёт requests для дубликатов                    |
| `overwriteDuplicates: true`  | генерация выполняется для всех пар заново                    |

### Прокси

| Факт              | Детали                                                                                                             |
| ----------------- | ------------------------------------------------------------------------------------------------------------------ |
| Прокси глобальный | `ai.proxy.enabled=true` → оба провайдера через прокси, `false` → оба напрямую. Per-request переключение невозможно |

## 4. Удаление типов и стилей

| Факт                           | Детали                                                                  |
| ------------------------------ | ----------------------------------------------------------------------- |
| Удалить может только создатель | проверяется локально (createdByUserId известен из загруженных данных)   |
| Перед удалением                | assets и generation_requests переназначаются на id=1 ("Неопределённый") |
| `created_by_user_id` nullable  | при удалении пользователя → SET NULL, тип/стиль остаётся без автора     |

## 5. Избранное

| Факт                      | Детали                                                                                |
| ------------------------- | ------------------------------------------------------------------------------------- |
| Переключение звёздочкой   | toggle на карточке типа/стиля, локальное состояние синхронизируется с сервером        |
| Серверная идемпотентность | повторное добавление → `ON CONFLICT DO NOTHING`, удаление несуществующего → тихий 200 |

## 6. Заголовок `UserId`

| Факт                      | Детали                                                                      |
| ------------------------- | --------------------------------------------------------------------------- |
| Используется в            | POST /generations, POST/DELETE /styles, POST/DELETE /image-types, favorites |
| Бэк не проверяет          | существование пользователя с этим id                                        |
| При несуществующем userId | FK-ошибка при записи в БД → 400 "Нарушение ограничений базы данных"         |

## 7. Ответы на ошибки

| HTTP | Источник                          | Тело                                                           |
| ---- | --------------------------------- | -------------------------------------------------------------- |
| 400  | `IllegalArgumentException`        | `{"error": "текст проверки"}` — конкретное сообщение           |
| 400  | `DataIntegrityViolationException` | `{"error": "Нарушение ограничений базы данных"}` — без деталей |
| 403  | `IllegalStateException`           | `{"error": "текст"}`                                           |
| 404  | `EntityNotFoundException`         | `{"error": "текст"}`                                           |
| 502  | `AiProviderException`             | `{"error": "Ошибка AI-провайдера: сообщение"}`                 |
| 500  | `Exception` (всё остальное)       | `{"error": "сырое сообщение исключения"}` — утечка деталей     |

- При 400 от `DataIntegrityViolation` — невозможно понять из ответа, какое именно поле нарушено.

## 8. Галерея

| Факт               | Детали                                                                                  |
| ------------------ | --------------------------------------------------------------------------------------- |
| Загрузка assets    | `GET /assets?imageTypeId=X&styleId=Y` — полный список без пагинации                     |
| Фильтрация         | обязательно выбрать тип (левая панель), стиль опционален (radio-кнопки)                 |
| Отображение        | `LazyVerticalGrid`, адаптивные колонки (min 180dp)                                      |
| Изображения        | placeholder-иконки, загрузка реальных картинок не реализована                           |
| Просмотр           | клик по карточке → lightbox-диалог (id, стиль, дата)                                    |
| Действия           | нет удаления, скачивание на устройство - при клике на появляющуюся при наведении кнопку |
| Поиск типов/стилей | case-insensitive подстрока по имени                                                     |
| id=1 в стилях      | скрыт из фильтра, в типах отображается серым (disabled)                                 |

## 9. Пагинация

| Эндпоинт       | Пагинация | Обоснование                                                                        |
| -------------- | --------- | ---------------------------------------------------------------------------------- |
| `/assets`      | да        | количество записей растёт с каждой генерацией — без пагинации весь список в памяти |
| `/image-types` | нет       | десятки записей максимум, пользователь создаёт вручную — сотен не будет            |
| `/styles`      | нет       | аналогично типам                                                                   |

**Поведение клиента для `/assets`:**

- `LazyVerticalGrid` подгружает следующую страницу при прокрутке к концу списка
- Размер страницы — 20 записей
- Первая страница загружается при выборе типа/стиля в фильтре
