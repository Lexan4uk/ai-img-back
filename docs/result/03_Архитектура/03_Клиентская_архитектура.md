# 03. Клиентская архитектура

## Общие сведения

Клиент — десктопное приложение на **Compose Multiplatform** (JVM target).
Язык интерфейса — русский. Собирается Gradle.

---

## Стек

| Компонент             | Технология                                 |
| --------------------- | ------------------------------------------ |
| UI-фреймворк          | Compose Multiplatform Desktop, Material 3  |
| Навигация             | Voyager (Tab + Navigator)                  |
| DI                    | Koin (3 модуля)                            |
| Управление состоянием | Kotlin `StateFlow` / `MutableStateFlow`    |
| ScreenModels          | Voyager `ScreenModel` + `screenModelScope` |
| Асинхронность         | Kotlin Coroutines (`Dispatchers.IO`)       |
| Настройки             | Java Preferences API (`JvmAppPreferences`) |
| Логирование           | Logback                                    |

## Архитектура

Используется паттерн на базе **Voyager ScreenModel** — аналог MVVM, где ScreenModel хранит состояние экрана, а Composable-функции рисуют UI по `StateFlow`.

```
┌───────────────────────────────────────┐
│         View (Compose UI)              │
│  @Composable функции                  │
│  Читают state из ScreenModel          │
│  Вызывают методы ScreenModel          │
├───────────────────────────────────────┤
│         ScreenModel (Voyager)          │
│  MutableStateFlow → StateFlow         │
│  Бизнес-логика экрана                 │
│  Coroutine scope: screenModelScope    │
├───────────────────────────────────────┤
│         Repository (интерфейс)         │
│  Сейчас: Mock (в памяти)             │
│  Будет: HTTP через client-utils       │
└───────────────────────────────────────┘
```

### Управление состоянием

Каждый ScreenModel хранит состояние в `MutableStateFlow` и выставляет наружу `StateFlow`:

```kotlin
private val _state = MutableStateFlow(GenerationState())
val state: StateFlow<GenerationState> = _state.asStateFlow()

fun toggleType(id: Long) {
    _state.update { it.copy(checkedTypeIds = ...) }
}
```

Асинхронные операции (загрузка данных, генерация) выполняются в `screenModelScope.launch { withContext(Dispatchers.IO) { ... } }`.

---

## ScreenModel-ы

| ScreenModel             | Scope     | Назначение                                                |
| ----------------------- | --------- | --------------------------------------------------------- |
| `GalleryScreenModel`    | Factory   | Фильтрация по типу/стилю, загрузка assets, избранное      |
| `GenerationScreenModel` | Factory   | Форма генерации, выбор модели, check/generate, результаты |
| `SettingsScreenModel`   | Singleton | URL сервера, userId, тема. Сохраняет в `AppPreferences`   |

ScreenModel-ы инжектируются через Koin: `val screenModel = koinScreenModel<GalleryScreenModel>()`.

### Синхронизация данных между экранами

`GalleryScreenModel` и `GenerationScreenModel` зарегистрированы как **factory** — при каждом переходе на вкладку создаётся новый экземпляр, который заново загружает данные из репозиториев. Общего состояния между экранами нет: нет shared `StateFlow`, event bus или кэша.

| Действие                | Экран «Генерация»                                              | Экран «Галерея»                                                                       |
| ----------------------- | -------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| Создание типа/стиля     | появляется сразу (StateFlow обновляется в текущем ScreenModel) | появится при переходе на вкладку (новый factory-экземпляр загрузит актуальный список) |
| Удаление типа/стиля     | аналогично                                                     | аналогично                                                                            |
| Переключение избранного | аналогично                                                     | аналогично                                                                            |

`SettingsScreenModel` — singleton, его состояние (URL сервера, userId, тема) доступно всегда и не пересоздаётся при навигации.

## Навигация (Voyager)

Навигация построена на Voyager `TabNavigator`. Два экрана — `GalleryTab` и `GenerationTab` — реализованы как singleton-объекты, имплементирующие `Tab`:

```kotlin
object GalleryTab : Tab {
    override val options @Composable get() = TabOptions(...)
    @Composable override fun Content() { ... }
}
```

`App.kt` оборачивает контент в `TabNavigator(GalleryTab) { Navigator(TabContentScreen) { ... } }`.

Sidebar содержит 2 пункта: Галерея и Генерация. Состояние sidebar (свёрнута/развёрнута) сохраняется в `AppPreferences`.

## Mock-репозитории

Все репозитории реализованы как mock-и с данными в памяти. HTTP-вызовов к серверу нет.

| Репозиторий                | Начальные данные                         | Особенности                            |
| -------------------------- | ---------------------------------------- | -------------------------------------- |
| `MockImageTypeRepository`  | 5 типов (включая id=1 «Неопределённый»)  | CRUD, избранное, `nextId` counter      |
| `MockStyleRepository`      | 6 стилей (включая id=1 «Неопределённый») | CRUD, избранное, `nextId` counter      |
| `MockAssetRepository`      | ~30 записей с пустыми `fileUri`          | Фильтрация по typeId/styleId           |
| `MockGenerationRepository` | —                                        | `ErrorMode`: SUCCESS / PARTIAL / CRASH |
| `MockUserRepository`       | 3 пользователя                           | Только чтение                          |

`MockGenerationRepository` поддерживает режимы ошибок для тестирования UI:

- **SUCCESS** — все запросы успешны
- **PARTIAL** — каждый 3-й запрос FAILED с рандомным сообщением
- **CRASH** — выбрасывает `"Connection refused: localhost:8080"`

Для перехода на реальный сервер — заменить mock-реализации на HTTP-классы в `repositoryModule` Koin. Интерфейсы репозиториев останутся без изменений.

## AiModelRegistry

Реестр моделей — singleton-объект `AiModelRegistry` в `core/AiModelConfig.kt`. Содержит три модели:

| ID в реестре              | provider    | model              | Aspect ratios                                   |
| ------------------------- | ----------- | ------------------ | ----------------------------------------------- |
| `openai-gpt-image-1-mini` | `openai`    | `gpt-image-1-mini` | 1:1, 1:1.5, 1.5:1, auto                         |
| `openai-gpt-image-1.5`    | `openai`    | `gpt-image-1.5`    | 1:1, 1:1.5, 1.5:1, auto                         |
| `stability-core`          | `stability` | `core`             | 1:1, 16:9, 9:16, 3:2, 2:3, 4:5, 5:4, 21:9, 9:21 |

Каждая модель (`AiModelConfig`) содержит: `id`, `provider`, `model`, `displayName`, `aspectRatios`, `defaultAspectRatio`.

- При смене модели в dropdown — список aspect ratio обновляется
- Если текущий ratio не поддерживается новой моделью — сбрасывается на `defaultAspectRatio`
- Реестр существует **только на клиенте**. Бэкенд не валидирует (подробнее — [05*Интеграция*с_AI.md](05_Интеграция_с_AI.md), §9)

## Обработка ошибок

### Ошибки генерации

Обрабатываются в `GenerationScreenModel.generate()`:

| Сценарий                            | Что видит пользователь                                               |
| ----------------------------------- | -------------------------------------------------------------------- |
| Сервер вернул результаты с ошибками | Экран результатов: ✅ успешные, ❌ с текстом ошибки (поштучно)       |
| Сервер полностью недоступен         | Экран результатов: ❌ все пары как FAILED (синтетические результаты) |
| Ошибка создания типа/стиля          | AlertDialog с текстом ошибки и кнопкой «OK»                          |
| Ошибка загрузки справочников        | AlertDialog с текстом ошибки                                         |

**Синтетические результаты:** когда HTTP-запрос полностью проваливается (таймаут, connection refused), клиент самостоятельно формирует `List<GenerationResult>` — по одному FAILED-результату на каждую запланированную пару (тип × стиль).

### Общий паттерн

```kotlin
// ScreenModel: state содержит error: String?
// UI: если error != null → AlertDialog
// Dismiss: screenModel.dismissError() очищает ошибку
```

## Тема

Material 3 с тремя режимами: **System**, **Light**, **Dark** (enum `ThemeMode`).

Переключение — кнопка в Header. Выбор сохраняется в `AppPreferences` и восстанавливается при перезапуске.
