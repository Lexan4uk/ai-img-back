# 02. Серверная архитектура

## Общая схема

```
ai-img-back (parent POM, packaging: pom)
│
├── client-utils/          ← Общий модуль (DTO, интерфейсы, HTTP-клиент)
│   └── ai-img-client-utils.jar
│
└── web/                   ← Spring Boot приложение
    └── ai-img-web.jar (executable)
```

Сервер состоит из **двух Maven-модулей** в одном репозитории (monorepo).
Модуль `client-utils` — библиотека без Spring Boot runtime. Модуль `web` —
полноценное Spring Boot приложение, которое зависит от `client-utils`.

## Модуль client-utils

Общий контракт между сервером и клиентом: DTO, интерфейсы контроллеров, HTTP-клиент.

```
client-utils/src/main/java/
└── com.example.ai_img_back.clientutils/
    ├── ClientUtil.java                 ← HTTP-клиент (OkHttp + Proxy)
    ├── controller/                     ← Интерфейсы контроллеров
    │   ├── IUserController.java
    │   ├── IImageTypeController.java
    │   ├── IStyleController.java
    │   ├── IAssetController.java
    │   └── IGenerationController.java
    ├── dto/                            ← Data Transfer Objects
    │   ├── UserDTO.java / UserRequest.java
    │   ├── ImageTypeDTO.java / ImageTypeRequest.java
    │   ├── StyleDTO.java / StyleRequest.java
    │   ├── AssetDTO.java
    │   ├── GenerateRequest.java
    │   ├── GenerationCheckResult.java  ← Результат POST /generations/check
    │   └── GenerationResultDTO.java
    └── enums/
        └── RequestStatus.java          ← PENDING / RUNNING / DONE / FAILED
```

### Зависимости client-utils

| Зависимость             | Причина                                                   |
| ----------------------- | --------------------------------------------------------- |
| spring-boot-starter-web | Аннотации Spring MVC (@GetMapping, @RequestMapping и др.) |
| OkHttp 4.12             | Кроссплатформенный HTTP-клиент (JVM + Android)            |
| Lombok                  | @Getter, @Setter, @Accessors                              |

client-utils **не запускает** Spring Boot — использует spring-web **только ради аннотаций** маппинга, которые парсит `ClientUtil` через рефлексию.

## Слои (Layered Architecture)

```
┌───────────────────────────────────────────────┐
│               Controller Layer                 │
│  UserController, ImageTypeController, ...      │
│  Повторяет контракт интерфейсов client-utils   │
│  @RestController, @RequestMapping              │
├───────────────────────────────────────────────┤
│               Service Layer                    │
│  UserService, ImageTypeService,                │
│  StyleService, GenerationService               │
│  @Service (без @Transactional на классе)       │
│  Бизнес-логика, валидация, оркестрация         │
├───────────────────────────────────────────────┤
│              Repository Layer                  │
│  UserRepository, AssetRepository, ...          │
│  @Repository                                   │
│  Чистый SQL через NamedParameterJdbcTemplate   │
│  RowMapper для маппинга ResultSet → Entity     │
├───────────────────────────────────────────────┤
│              PostgreSQL Database               │
│  Liquibase migrations                          │
│  8 таблиц, BIGSERIAL PK                       │
└───────────────────────────────────────────────┘
```

### Controller Layer

- Каждый контроллер **повторяет контракт** интерфейса из `client-utils`
  (те же URL, методы и типы, но без `implements` — интерфейсы используются
  только на клиенте для прокси через `ClientUtil`)
- Маппинг аннотациями Spring MVC: `@GetMapping`, `@PostMapping`, `@DeleteMapping`
- `UserId` приходит в заголовке запроса: `@RequestHeader("UserId") Long userId`
- Контроллеры тонкие — делегируют в сервисы
- `GenerationController` содержит два POST-эндпоинта: `/generations/check` и `/generations`

### Service Layer

- Бизнес-логика: валидация, правила удаления, генерация
- `GenerationService` — **без `@Transactional` на классе**. Каждый вызов `markRunning` / `markDone` / `markFailed` коммитится сразу через `jdbcTemplate` (auto-commit). Причина: `generate()` и `resumeBatchAsync()` выполняются минуты (вызовы AI по 8–30 сек каждый). Открытая транзакция блокировала бы соединение и откатывала уже завершённые запросы при сбое
- `@Async` на `resumeBatchAsync()` — фоновое возобновление batch при старте сервера
- Константы: `UNDEFINED_TYPE_ID = 1L`, `UNDEFINED_STYLE_ID = 1L`
- `GenerationService` — центральный сервис: `check()` (подсчёт дубликатов), `generate()` (основной цикл), `resumeBatchAsync()` (возобновление)

### Repository Layer

- **Не Spring Data JPA**. Все запросы — ручной SQL через `NamedParameterJdbcTemplate`
- `INSERT ... RETURNING *` — PostgreSQL возвращает созданную строку
- `RowMapper<Entity>` — лямбда-маппер `ResultSet → Entity`
- `MapSqlParameterSource` — именованные параметры (`:imageTypeId`, `:hash`)

**Пример паттерна (AssetRepository):**

```java
@Repository
@RequiredArgsConstructor
public class AssetRepository {

    private final NamedParameterJdbcTemplate jdbcTemplate;

    private static final RowMapper<Asset> ASSET_MAPPER = (rs, rowNum) -> new Asset()
            .setId(rs.getLong("id"))
            .setImageTypeId(rs.getLong("image_type_id"))
            .setStyleId(rs.getLong("style_id"))
            .setFileUri(rs.getString("file_uri"))
            .setCreatedAt(rs.getObject("created_at", OffsetDateTime.class));

    public Asset create(Asset asset) {
        String sql = """
                INSERT INTO assets (image_type_id, style_id, file_uri)
                VALUES (:imageTypeId, :styleId, :fileUri)
                RETURNING *
                """;
        return jdbcTemplate.queryForObject(sql, params, ASSET_MAPPER);
    }
}
```

## Возобновление при рестарте

`GenerationStartupCleaner` — при `ApplicationReadyEvent` находит незавершённые batch (запросы со статусом RUNNING или PENDING) и вызывает `GenerationService.resumeBatchAsync()` для каждого. Обработка продолжается в фоне (`@Async`) — уже завершённые запросы пропускаются, оставшиеся обрабатываются заново. Не блокирует старт приложения.

Подробнее — [05*Интеграция*с_AI.md](05_Интеграция_с_AI.md), §12 (восстановление при рестарте).

## Обработка ошибок

`GlobalExceptionHandler` (`@RestControllerAdvice`) перехватывает исключения и возвращает единообразный JSON:

| Исключение                        | HTTP-код | Когда                                            |
| --------------------------------- | -------- | ------------------------------------------------ |
| `EntityNotFoundException`         | 404      | Сущность не найдена                              |
| `IllegalArgumentException`        | 400      | Ошибка валидации                                 |
| `DataIntegrityViolationException` | 400      | Нарушение constraint БД                          |
| `IllegalStateException`           | 403      | Запрещённая операция (чужой ресурс, неудаляемый) |
| `AiProviderException`             | 502      | Ошибка AI-провайдера                             |
| `Exception`                       | 500      | Всё остальное                                    |

Формат ответа: `{ "error": "текст ошибки" }`

`AiProviderException` имеет отдельный обработчик → 502 Bad Gateway. В штатном потоке она перехватывается внутри `GenerationService`, обработчик добавлен как защитная сетка.

Подробнее — [01_Бэкенд.md](../04_Жизненный_Цикл/01_Бэкенд.md), §8.

## Тестирование

Интеграционные end-to-end тесты.

```
web/src/test/java/
└── com.example.ai_img_back/
    ├── BaseTest.java                  ← Базовый класс (Testcontainers, TestRestTemplate)
    ├── UserControllerTest.java
    ├── ImageTypeControllerTest.java
    ├── StyleControllerTest.java
    ├── AssetControllerTest.java
    └── GenerationControllerTest.java
```

- `BaseTest` — базовый класс: `@SpringBootTest(webEnvironment = RANDOM_PORT)`, настройка **Testcontainers** (реальный PostgreSQL в Docker)
- Запросы через `TestRestTemplate` к реальному серверу
- Liquibase автоматически накатывает миграции на тестовую БД
- Покрытие: CRUD-операции, избранное, генерация, дедупликация (`check` + `generate`)

## Граф зависимостей модулей

```
┌──────────────────────┐
│    ai-img-back       │  (parent POM)
│    packaging: pom    │
└──────────┬───────────┘
           │
     ┌─────┴──────┐
     ▼            ▼
┌──────────┐  ┌──────────┐
│ client-  │  │   web    │
│  utils   │◄─│ (Spring  │
│  (.jar)  │  │  Boot)   │
└──────────┘  └──────────┘
     ▲
     │  Maven dependency
     │
┌────┴──────────────────┐
│  aiimageclient        │  (Compose Desktop, Gradle)
└───────────────────────┘
```

`client-utils` используется как общий контракт: `web` подключает его напрямую, `aiimageclient` — через `ClientUtil` (OkHttp-прокси, который строит HTTP-запросы по аннотациям интерфейсов контроллеров).
