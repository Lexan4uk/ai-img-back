# 04. Взаимодействие клиент-сервер

## Общая схема

```
┌──────────────────────────┐          ┌──────────────────────────┐
│      Desktop Client      │          │       Spring Boot         │
│    (Compose / Kotlin)    │   HTTP   │        Server             │
│                          │ ◄──────► │                          │
│  ServerConnectionManager │          │  Controllers              │
│        ↓                 │          │      ↓                    │
│    ClientUtil            │          │  Services                 │
│        ↓                 │          │      ↓                    │
│  IImageTypeController    │          │  Repositories             │
│   (OkHttp Proxy)         │          │      ↓                    │
│                          │          │  PostgreSQL               │
└──────────────────────────┘          └──────────────────────────┘
```

Клиент и сервер **используют один и тот же модуль** `client-utils`:

- **Сервер** реализует интерфейсы (`UserController implements IUserController`)
- **Клиент** вызывает те же интерфейсы через прокси (`ClientUtil.getClient(IUserController.class)`)

Результат: **единый контракт API**, гарантированная совместимость типов.

---

## ClientUtil — механизм прокси

`ClientUtil` — ядро взаимодействия. Он превращает вызов метода Java-интерфейса
в HTTP-запрос через `java.lang.reflect.Proxy`.

### Пошаговый алгоритм

```
Клиент вызывает:  ctrl.getAll()
                       │
          ┌────────────┴────────────┐
          │   java.lang.reflect     │
          │       .Proxy            │
          │   InvocationHandler     │
          └────────────┬────────────┘
                       │
    ┌──────────────────┴──────────────────┐
    │  ШАГ 1: Определить HTTP-метод       │
    │  @GetMapping → "GET"                │
    │  @PostMapping → "POST"              │
    │  @DeleteMapping → "DELETE"          │
    ├─────────────────────────────────────┤
    │  ШАГ 2: Разобрать параметры         │
    │  @PathVariable → подстановка в URL  │
    │  @RequestParam → query-параметры    │
    │  @RequestBody → JSON тело           │
    │  @ModelAttribute → поля → query     │
    ├─────────────────────────────────────┤
    │  ШАГ 3: Собрать URL                 │
    │  baseUrl + controllerPath +         │
    │  methodPath + queryParams           │
    ├─────────────────────────────────────┤
    │  ШАГ 4: Выполнить HTTP-запрос       │
    │  OkHttp → Request → execute()       │
    ├─────────────────────────────────────┤
    │  ШАГ 5: Десериализовать ответ       │
    │  Jackson ObjectMapper →             │
    │  method.getGenericReturnType()      │
    └─────────────────────────────────────┘
                       │
                       ▼
              Результат: List<ImageTypeDTO>
```

### Пример трансформации

Определение в интерфейсе:

```java
@RequestMapping("/image-types")
public interface IImageTypeController {

    @GetMapping
    List<ImageTypeDTO> getAll();

    @DeleteMapping("/{id}")
    void delete(@PathVariable Long id,
                @RequestHeader("UserId") Long userId);

    @PostMapping("/{id}/favorite")
    void addFavorite(@PathVariable Long id,
                     @RequestHeader("UserId") Long userId);
}
```

Вызов на клиенте:

```kotlin
val ctrl = connectionManager.getController(IImageTypeController::class.java)

// getAll() → GET http://localhost:8080/image-types
val types: List<ImageTypeDTO> = ctrl.getAll()

// delete(5) → DELETE http://localhost:8080/image-types/5
ctrl.delete(5L, userId)

// addFavorite(3) → POST http://localhost:8080/image-types/3/favorite
ctrl.addFavorite(3L, userId)
```

> **Примечание:** `@RequestHeader("UserId")` на клиенте игнорируется —
> UserId добавляется автоматически через OkHttp Interceptor.

## Авторизация

Текущая реализация — **заглушка вместо JWT**:

```
┌───────────┐                    ┌──────────┐
│  Клиент   │  UserId: 2         │  Сервер  │
│           │ ──────────────────► │          │
│           │  Header: UserId     │          │
└───────────┘                    └──────────┘
```

- UserId передаётся как **HTTP-заголовок** `UserId` (Long)
- OkHttp Interceptor в `ClientUtil` добавляет его ко **всем** запросам
- Сервер читает: `@RequestHeader("UserId") Long userId`
- В продакшене → замена на JWT-токен

## Формат данных

### Запросы

| Тип    | Content-Type     | Тело                |
| ------ | ---------------- | ------------------- |
| GET    | —                | — (параметры в URL) |
| POST   | application/json | JSON-объект         |
| DELETE | —                | —                   |

### Ответы

| Код | Формат | Пример                                  |
| --- | ------ | --------------------------------------- |
| 200 | JSON   | `[{"id": 1, "name": "Фото", ...}]`      |
| 400 | JSON   | `{"error": "Имя не может быть пустым"}` |
| 403 | JSON   | `{"error": "Нельзя удалить чужой тип"}` |
| 404 | JSON   | `{"error": "Тип не найден"}`            |
| 500 | JSON   | `{"error": "Internal error"}`           |

### Обработка ошибок на клиенте

```kotlin
try {
    val types = ctrl.getAll()
} catch (e: RuntimeException) {
    // ClientUtil бросает RuntimeException с телом ответа
    // e.message = "HTTP 404" или "Тип не найден"
    showSnackbar(e.message ?: "Неизвестная ошибка", SnackbarType.ERROR)
}
```

## Узкие места взаимодействия (v1)

### Блокирующий вызов генерации

`POST /generations` — **синхронный** эндпоинт. Клиент отправляет запрос и ждёт,
пока сервер обработает **все** пары тип × стиль. На время генерации:

- Клиент **заблокирован** (показывает спиннер без прогресса)
- HTTP-соединение **открыто** на протяжении всей генерации (может быть 1–50 минут)
- **Нет прогресс-бара** — пользователь не знает, сколько картинок уже готово
- **Нет отмены** — невозможно прервать генерацию, не закрыв приложение
- **Нет таймаута** на клиенте — если сервер зависнет, клиент будет ждать бесконечно

```
Клиент                          Сервер
  │                                │
  │  POST /generations             │
  │  (3 типа × 2 стиля = 6)       │
  │ ──────────────────────────►    │
  │                                │── AI запрос 1 (8-30 сек)
  │        тишина...               │── AI запрос 2 (8-30 сек)
  │        тишина...               │── AI запрос 3 (8-30 сек)
  │        тишина...               │── AI запрос 4 (8-30 сек)
  │        тишина...               │── AI запрос 5 (8-30 сек)
  │        тишина...               │── AI запрос 6 (8-30 сек)
  │                                │
  │  200 OK [6 results]            │
  │ ◄──────────────────────────    │
  │                                │
  │  Общее время: 48-180 сек      │
```

### Поведение при обрыве соединения

| Сценарий                     | Что видит пользователь                                        |
| ---------------------------- | ------------------------------------------------------------- |
| Сервер вернул результаты     | Экран результатов (✅/❌/⏭ по каждой паре)                   |
| Сервер упал / таймаут / сеть | Экран результатов со **всеми** парами как ❌ FAILED            |
| Отдельный AI-вызов упал      | Результаты: часть ✅, часть ❌ (поштучно)                     |

> При полном сбое соединения клиент создаёт **синтетические результаты** —
> по одному FAILED на каждую планировавшуюся пару тип × стиль, с текстом ошибки.

**Решение (v2):** асинхронный эндпоинт + WebSocket/SSE для real-time прогресса:

```
POST /generations  → 202 Accepted, { "batchId": 42 }
WS /generations/42/progress → { "done": 3, "total": 6, "lastResult": {...} }
GET /generations/42/results → полный результат
```

## Жизненный цикл подключения

```
┌──────────┐     ┌───────────────┐     ┌──────────┐
│ Запуск   │────►│  Настройки    │────►│  Работа  │
│ клиента  │     │  (URL, User)  │     │  с API   │
└──────────┘     └───────┬───────┘     └──────────┘
                         │
                         ▼
                  ┌──────────────┐
                  │  Проверка    │
                  │  подключения │
                  │  GET /users  │
                  └──────┬───────┘
                         │
                    ┌────┴────┐
                    │         │
                 Успех     Ошибка
                    │         │
                    ▼         ▼
              ● Подключён  ● Отключён
              (зелёный)    (красный)
```

1. При запуске — загрузка URL и UserId из настроек
2. Проверка подключения — тестовый запрос `GET /users`
3. Успех → footer показывает зелёный статус
4. Ошибка → красный статус, работа в offline (если есть кеш)
5. Смена URL/UserId → пересоздание `ClientUtil`
