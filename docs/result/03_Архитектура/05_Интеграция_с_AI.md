# 05. Интеграция с AI-провайдерами

## Общая картина

Система генерирует изображения по текстовому описанию пользователя.
Подключены два AI-провайдера — **OpenAI** и **Stability AI**. Картинка сохраняется на диск сервера и регистрируется в базе данных.

```
                                         ┌───────────────────────┐
                                    ┌───►│  OpenAI                │
                                    │    │  gpt-image-1-mini      │
┌────────┐   ┌──────────┐   ┌──────┴──┐ │  gpt-image-1.5         │
│ Клиент │──►│ Генера-  │──►│ Provider │ └───────────────────────┘
│        │   │   ция    │   │  Router  │
└────────┘   └──────────┘   └──────┬──┘ ┌───────────────────────┐
                                    │    │  Stability AI          │
                                    └───►│  Stable Image Core     │
                                         │  (модель из конфига)   │
                                         └───────────────────────┘
```

> **Язык промптов:** Stability AI **не понимает русский** — промпты должны быть на английском. OpenAI понимает русский, но качество выше на английском.

> **Прокси:** OpenAI заблокирован в России — нужен прокси. Подробности — см. [06_Прокси.md](06_Прокси.md).

## 1. OpenAI GPT Image

### Что это

Сервис генерации картинок от OpenAI. Доступны две модели: `gpt-image-1-mini` (быстрее и дешевле) и `gpt-image-1.5` (выше качество). Клиент выбирает модель при отправке запроса — поле `model` в DTO. Если не указана, сервер использует модель из конфига (`ai.openai.model`, по умолчанию `gpt-image-1.5`).

### Характеристики

| Параметр         | Значение                                              |
| ---------------- | ----------------------------------------------------- |
| Модели           | `gpt-image-1-mini`, `gpt-image-1.5`                   |
| Стоимость        | ~$0.02–0.19 / картинка (зависит от качества/размера)  |
| Скорость         | ~8–30 секунд (зависит от модели)                      |
| Доступ из России | Нет — нужен прокси (см. [06_Прокси.md](06_Прокси.md)) |
| Язык промптов    | Русский и английский (на английском качество выше)    |
| Формат ответа    | JSON с base64-строкой                                 |

### Как работает запрос

**Протокол:** `application/json`

```
POST https://api.openai.com/v1/images/generations
Content-Type: application/json
Authorization: Bearer <api-key>

{
  "model": "gpt-image-1.5",
  "prompt": "<finalPrompt>",
  "n": 1,
  "size": "1024x1024",
  "output_format": "png",
  "quality": "high"
}
```

- `quality: "high"` — захардкожен, всегда отправляется
- `n: 1` — всегда одна картинка за запрос (подробнее — §6 «Генерация батчами»)
- `output_format: "png"` — формат выходного изображения

**Ответ (HTTP 200):**

```json
{
	"data": [{ "b64_json": "<base64-encoded-png>" }]
}
```

Обработка: извлекается `data[0].b64_json` → `Base64.getDecoder().decode(b64)` → `byte[]`. Байты передаются дальше в `ImageStorageService` для сохранения на диск.

### Aspect ratio

OpenAI принимает размер в пикселях (`size`). Наш сервер конвертирует строковый `aspectRatio` из запроса клиента в формат OpenAI:

| aspectRatio (от клиента) | size (отправляется в OpenAI) | Ориентация |
| ------------------------ | ---------------------------- | ---------- |
| `1:1`                    | `1024x1024`                  | Квадрат    |
| `1:1.5`                  | `1024x1536`                  | Портрет    |
| `1.5:1`                  | `1536x1024`                  | Альбом     |
| `null`, `""`, другое     | `auto`                       | Авто       |

Всего **3 фиксированных размера** + `auto`. Если клиент передаёт соотношение, не совпадающее ни с одним из трёх — система выставляет `auto`.

### Особенности

- Возвращает **base64** — декодирование происходит в памяти. Для одного изображения это ~2–5 MB
- Поддерживает **выбор модели** — клиент может указать `gpt-image-1-mini` или `gpt-image-1.5`
- Требует **прокси** из России (`ai.proxy.enabled=true`)
- Параметр `quality: "high"` зафиксирован — другие варианты не передаются

### Асинхронные механизмы (не используются)

Наш код вызывает API **синхронно** — отправил запрос, ждёшь ответ с картинкой. OpenAI предоставляет три альтернативных способа, которые мы не используем:

| Механизм                            | Суть                                                             | Latency        | Стоимость             |
| ----------------------------------- | ---------------------------------------------------------------- | -------------- | --------------------- |
| **Batch API** (`/v1/batches`)       | JSONL-файл с запросами (до 50 000), результат в течение 24 часов | до 24 ч        | **−50%**              |
| **Background Mode** (Responses API) | `background: true` → polling по ID                               | секунды–минуты | стандартная           |
| **Streaming** (SSE)                 | Промежуточные превью (`partial_images: 0–3`)                     | реалтайм       | +100 tokens / partial |

Для текущего масштаба (десятки картинок за раз) синхронный подход оптимален.

## 2. Stability AI

### Что это

Сервис генерации картинок от Stability AI. Работает напрямую из России без прокси. Модель определяет endpoint API — значение `model` подставляется в конец URL. По умолчанию — `core` (Stable Image Core).

### Характеристики

| Параметр         | Значение                         |
| ---------------- | -------------------------------- |
| Модель           | `core` (Stable Image Core)       |
| Стоимость        | 3 кредита за генерацию (~$0.03)  |
| Скорость         | ~5–10 секунд                     |
| Разрешение       | 1.5 мегапикселя                  |
| Доступ из России | Да — работает напрямую           |
| Язык промптов    | **Только английский**            |
| Формат ответа    | Сырые байты (не JSON, не base64) |

### Как работает запрос

**Протокол:** `multipart/form-data` — принципиально отличается от OpenAI

URL конструируется из базового адреса и модели: `ai.stability.url` + `"/"` + `model`.

```
POST https://api.stability.ai/v2beta/stable-image/generate/core
                                                           ^^^^
                                                     ai.stability.model
Content-Type: multipart/form-data
Authorization: Bearer <api-key>
Accept: image/*

Parts:
  prompt = "<finalPrompt>"
  output_format = "png"
  aspect_ratio = "1:1"
```

Если клиент не указал модель — берётся `ai.stability.model` из конфига (по умолчанию `core`).

**Ответ (HTTP 200):**

Тело ответа — **сырые байты** изображения (PNG/WebP/JPEG в зависимости от `output_format`). Не JSON, не base64. Сервер вызывает `response.getBody()` → `byte[]` и передаёт на сохранение.

### Aspect ratio

Stability принимает строковое соотношение сторон (`aspect_ratio`). Допустимые значения фиксированы — если клиент передаёт нестандартное, сервер конвертирует в ближайшее допустимое:

| Допустимый ratio | Ориентация      |
| ---------------- | --------------- |
| `1:1`            | Квадрат         |
| `16:9`           | Широкий альбом  |
| `9:16`           | Высокий портрет |
| `21:9`           | Ультраширокий   |
| `9:21`           | Ультравысокий   |
| `2:3`            | Портрет         |
| `3:2`            | Альбом          |
| `4:5`            | Портрет         |
| `5:4`            | Альбом          |

Всего **9 вариантов**. Алгоритм конвертации: разбить входной ratio на два числа, вычислить пропорцию `w/h`, найти допустимый ratio с минимальным отклонением. Примеры: `"1:1.5"` → `"2:3"`, `"1.5:1"` → `"3:2"`.

Если `aspectRatio` не указан (`null`, `""`) — используется `"1:1"`.

### Особенности

- Возвращает **сырые байты** — нет декодирования base64, меньше нагрузки на память
- Модель определяет **endpoint** — значение подставляется в URL (`ai.stability.url/core`)
- **3 кредита** за каждую генерацию
- **Только английский** в промптах — русский не обрабатывается
- Работает **напрямую** из России

### Асинхронные механизмы (отсутствуют)

API генерации у Stability **полностью синхронный**. Нет job ID, polling, вебхуков, batch API или streaming. Единственные async-эндпоинты Stability — **Creative Upscale** и **Image-to-Video**, но к генерации картинок они не относятся.

## 4. Stub-провайдер (для тестов)

Активируется через `ai.stub.enabled=true`.

- `providerName()` → `"stub"`
- `isAvailable()` → всегда `true`
- `generate()` → возвращает захардкоженный 1x1 красный PNG, дополненный до 1100 байт (чтобы пройти валидацию `MIN_IMAGE_SIZE=1024`)
- Не делает HTTP-запросов, мгновенный ответ, не кидает ошибок

## 5. Размеры изображений (aspect ratio)

Пользователь выбирает **соотношение сторон** (aspect ratio), а не пиксели.
Система конвертирует ratio в формат, который понимает конкретный провайдер.

**Пример:** клиент отправляет `"aspectRatio": "1:1.5"`:

- OpenAI получит `size: "1024x1536"` (портрет)
- Stability получит `aspect_ratio: "2:3"` (ближайшее допустимое)

Если aspect ratio не указан — используется **1:1** (квадрат) по умолчанию.

## 6. Генерация батчами (параметр `n` / `samples`)

Наш код всегда отправляет **по одной картинке за HTTP-запрос**: `"n": 1` для OpenAI, без параметра `samples` для Stability. Один `GenerationRequest` = один HTTP-вызов = одна картинка.

### Что поддерживают API

| Провайдер                                             | Параметр | Поддержка | Диапазон |
| ----------------------------------------------------- | -------- | --------- | -------- |
| OpenAI (gpt-image-1.5, gpt-image-1, gpt-image-1-mini) | `n`      | Да        | 1–10     |
| Stability (v2beta, `core`)                            | —        | **Нет**   | Всегда 1 |

При `n > 1` OpenAI возвращает массив `data[]` с несколькими `b64_json`. Цена линейная: 3 картинки стоят ×3, скидки за батч нет.

Stability v2beta **не поддерживает** batch-генерацию. Параметр `samples` существовал в deprecated v1 API (SDXL, SD 1.6), в текущем v2beta его нет.

### Почему не применимо в нашей архитектуре

Параметр `n` повторяет **один и тот же промпт** несколько раз. В нашей системе каждый `GenerationRequest` — это уникальная пара `(imageType, style)` со своим уникальным `finalPrompt`. Batch-параметр `n` генерирует варианты одного промпта, а не разных.

### Преимущества (если бы `n` подходил)

1. **Меньше HTTP-запросов** — вместо 10 запросов один. Меньше overhead на TLS handshake, заголовки, round-trip
2. **Проще код** — не нужен цикл на стороне сервера
3. **Атомарность** — все картинки одного промпта в одном ответе

### Недостатки

1. **Все-или-ничего при ошибке** — если запрос упал (таймаут, 5xx), теряются все картинки разом, а не одна. При `n=1` остальные уже сохранены
2. **Таймауты** — генерация 10 картинок ≈ 10× дольше. При `timeout-seconds=90` это почти гарантированный таймаут. Придётся ставить 300–600 с
3. **Память** — 10 base64-картинок в одном JSON ≈ 30–50 MB в памяти
4. **Несовместимость со Stability v2beta** — второй провайдер не поддерживает batch. Код пришлось бы раздваивать
5. **Retry усложняется** — при `n > 1` нужно разбираться, какие из batch прошли, а какие нет (если content filter сработал на 1 из 10)
6. **Resume при рестарте** — сейчас один request = один HTTP-вызов. Если batch-запрос с `n=5` упал на середине — нет возможности узнать, сколько картинок успело сгенериться на стороне API

## 7. Глобальный API-ключ

Система использует **один серверный API-ключ** на каждого провайдера. Ключи хранятся в `application.properties` / переменных окружения. Пользователи не предоставляют своих ключей — все запросы идут от имени сервера.

### Как это выглядит

```
Пользователь A ─┐
Пользователь B ─┼─► Сервер (один ключ OpenAI) ──► api.openai.com
Пользователь C ─┘
```

Все пользователи делят один ключ, один баланс, одни rate limits.

### Преимущества

1. **Пользователю не нужен аккаунт провайдера.** Не нужно регистрироваться в OpenAI/Stability, получать ключ, пополнять баланс. Порог входа минимальный
2. **Простая архитектура.** Один ключ в конфиге, один `RestTemplate` на провайдер. Нет хранения/ротации/валидации пользовательских ключей
3. **Контроль расходов.** Администратор видит общую стоимость на одном аккаунте провайдера. Не нужно агрегировать расходы с десятков пользовательских аккаунтов
4. **Безопасность ключей.** Ключи не передаются клиенту и не хранятся в пользовательских данных. Нет риска утечки через фронтенд или БД пользователей

### Недостатки

1. **Общий rate limit.** OpenAI ограничивает ~15 RPM (Tier 2) на аккаунт. Если три пользователя одновременно запустят генерацию — лимит будет превышен быстрее. Нет изоляции между пользователями
2. **Общий баланс.** Один пользователь может исчерпать баланс за всех. Нет механизма квот на пользователя
3. **Нет персонализации тарифов.** Нельзя дать одному пользователю доступ к дорогой модели, а другому — только к дешёвой. Все видят одинаковый набор моделей
4. **Каскадный отказ при проблемах с ключом.** Если ключ заблокирован или истёк — не работает ни у кого. 401/402/403 от провайдера останавливает весь batch и помечает оставшиеся запросы как FAILED
5. **Оператор платит за всё.** Стоимость генерации ложится на владельца сервера, а не на конечного пользователя

## 8. Настраиваемый шаблон промпта

Шаблон один на всю систему. Задача сервиса — генерация картинок по описанию, без редактирования, inpainting, img2img или других режимов. Один вход (текст) → один выход (картинка). Для этого достаточно одного шаблона, который склеивает тип, стиль и пользовательский промпт в финальную строку.

Шаблон единый для всех провайдеров и настраивается в `application.properties`:

```properties
ai.prompt-template=Create an image of type: {type}. Style: {style}. Description: {prompt}. Avoid unnecessary text in the image.
```

**Переменные:**

| Переменная | Что подставляется               | Пример                      |
| ---------- | ------------------------------- | --------------------------- |
| `{type}`   | Промпт типа изображения (из БД) | «Фотография», «Иллюстрация» |
| `{style}`  | Промпт стиля (из БД)            | «Минимализм», «Айвазовский» |
| `{prompt}` | Текст пользователя              | «Кузнец куёт меч у моря»    |

Итоговый промпт (`finalPrompt`) — одинаковый для любого провайдера. Каждая пара `(type, style)` в batch даёт свой уникальный `finalPrompt`.

## 9. Выбор модели на клиенте

Клиент предоставляет пользователю **dropdown выбор модели**. Реестр моделей хранится в `AiModelRegistry` на клиенте. Каждая модель определяет:

- Провайдер (`provider`) и имя модели (`model`) для бэкенда
- Допустимые **aspect ratio** (dropdown обновляется при смене модели)

### Доступные модели

| Модель в UI                 | provider    | model              | Aspect ratios                                   |
| --------------------------- | ----------- | ------------------ | ----------------------------------------------- |
| OpenAI GPT Image 1 Mini     | `openai`    | `gpt-image-1-mini` | 1:1, 1:1.5, 1.5:1, auto                         |
| OpenAI GPT Image 1.5        | `openai`    | `gpt-image-1.5`    | 1:1, 1:1.5, 1.5:1, auto                         |
| Stability Stable Image Core | `stability` | `core`             | 1:1, 16:9, 9:16, 3:2, 2:3, 4:5, 5:4, 21:9, 9:21 |

При смене модели клиент сбрасывает aspect ratio на значение по умолчанию, если текущий ratio не поддерживается новой моделью.

Для добавления новой модели — достаточно добавить запись в реестр.

### Валидация моделей

Реестр моделей существует **только на клиенте**. Бэкенд не хранит списка допустимых моделей и не проверяет значение `model` — он передаёт его провайдеру как есть. Для OpenAI значение уходит в поле `model` JSON-запроса, для Stability — подставляется в URL.

Если в `AiModelRegistry` указать несуществующую модель (например, `model: "xyz"`), бэкенд отправит запрос с этим значением и получит ошибку от провайдера: OpenAI вернёт 400, Stability — 404.

## 10. Дедупликация запросов

### Зачем

Каждая генерация стоит денег. Если один и тот же запрос отправить дважды — система обнаружит дубликат и позволит пользователю решить, платить повторно или нет.

### Как это работает

**Двухшаговый процесс:**

1. Клиент вызывает `POST /generations/check` — получает `{ totalCount, duplicateCount, newCount }`
2. Если есть дубликаты — клиент показывает диалог: «Генерация 20 картинок. 8 уже существуют. Пересоздать?»
3. Пользователь выбирает → клиент отправляет `POST /generations` с `overwriteDuplicates: true/false`
4. Если `false` — сервер **не создаёт** requests для дубликатов (они не попадают в БД, не занимают строку, не требуют обработки)
5. Если `true` — создаются requests для всех пар, включая дубликаты

### Хеш

Уникальный отпечаток запроса — SHA-256 хеш из итогового промпта и параметров генерации:

```
hash = SHA-256( finalPrompt + "|" + generationParams )
```

Если промпт, тип, стиль или aspect ratio отличаются хотя бы немного — хеш будет другим, и запрос считается новым.

| Изменение                                | Хеш изменится? | Дубликат? |
| ---------------------------------------- | -------------- | --------- |
| Другой текст промпта                     | Да             | Нет       |
| Другой тип изображения                   | Да             | Нет       |
| Другой стиль                             | Да             | Нет       |
| Другой aspect ratio                      | Да             | Нет       |
| Другой пользователь, всё остальное то же | Нет            | Да        |
| Всё одинаковое, повторный запрос         | Нет            | Да        |

### Жизненный цикл статусов

```
PENDING → RUNNING → DONE / FAILED
```

Дубликаты просто не создают запись в БД — нет отдельного статуса для пропущенных.

> Подробнее о хеше и дедупликации — см. [07_Генерация.md](../01_Модель_данных/07_Генерация.md).

## 11. Архитектура провайдеров в коде

### Паттерн Strategy

```
┌──────────────────────────────┐
│   ImageAiProvider             │  (интерфейс)
│   + generate(prompt,          │
│     aspectRatio, model)       │
│     → byte[]                  │
│   + providerName() → String   │
│   + isAvailable() → boolean   │
└─────────┬────────────────────┘
          │
   ┌──────┼────────────┐
   ▼                    ▼
┌────────────────┐ ┌──────────────────┐
│ OpenAiProvider  │ │StabilityProvider  │
│                 │ │                   │
│ JSON API        │ │ multipart/form    │
│ Bearer token    │ │ Bearer token      │
│ base64 ответ    │ │ байты напрямую    │
└────────────────┘ └──────────────────┘
```

Метод `generate` принимает 3 параметра: промпт, aspect ratio и **имя модели** (nullable — если null, используется модель по умолчанию для провайдера).

Все провайдеры возвращают `byte[]` — сырые байты изображения. Разница — в протоколе, формате запроса/ответа и обработке ошибок.

Добавление нового провайдера = один новый Java-класс, реализующий `ImageAiProvider`. Spring автоматически подхватит его.

### AiProviderException

```java
public class AiProviderException extends RuntimeException {
    int httpStatus;     // HTTP-код ответа (0 если не HTTP-ошибка)
    boolean retryable;  // можно ли повторить
}
```

- `retryable=true`: 5xx, 429 rate limit, сетевые ошибки (таймаут, отказ соединения, DNS)
- `retryable=false`: 400 bad request, 401 unauthorized, 402 payment required, 403 forbidden

### Логика выбора провайдера (ProviderRouter)

```
Пользователь указал provider?
├── Да → Использовать указанного
│        ├── Доступен? → Генерировать
│        └── Недоступен? → Ошибка
│
└── Нет (provider = null / "auto") → Автоматический выбор
    ├── Провайдер по умолчанию доступен? → Использовать его
    └── Иначе → Любой доступный
```

Когда пользователь явно указал провайдер — система **не переключается** на другой автоматически.

## 12. Обработка ошибок и retry

### Retry-стратегия

Каждый провайдер оборачивает HTTP-вызов в цикл retry с экспоненциальной задержкой:

```
Попытка 1 → ошибка → пауза 2 сек
Попытка 2 → ошибка → пауза 4 сек
Попытка 3 → ошибка → FAILED

Формула задержки: 2^attempt * 1000 мс
```

- OpenAI: до 3 попыток (`ai.openai.max-retries`)
- Stability: до 2 попыток (`ai.stability.max-retries`)

Retry **не делается** при: 400 (ошибка промпта), 401/403 (неверный ключ), 402 (нет баланса).

### Ошибки и их обработка

| Ситуация                   | Поведение                                                       |
| -------------------------- | --------------------------------------------------------------- |
| HTTP 400 Bad Request       | FAILED. Невалидный промпт/размер/модель. Retry не поможет       |
| HTTP 401/403 Unauthorized  | FAILED + `authFailed=true` → все оставшиеся в batch тоже FAILED |
| HTTP 402 Payment Required  | FAILED + `authFailed=true` → аналогично 401                     |
| HTTP 429 Rate Limit        | Retry через `2^attempt` секунд                                  |
| HTTP 5xx Server Error      | Retry                                                           |
| Таймаут / сетевая ошибка   | Retry                                                           |
| Прокси недоступен          | Retry (через `ResourceAccessException`)                         |
| Пустой ответ от провайдера | Retry                                                           |
| Промпт отклонён модерацией | FAILED для этого запроса. Остальные в batch продолжаются        |
| Диск заполнен              | FAILED. Картинка не сохранена                                   |

### Каскадная остановка при auth-ошибках

При получении 401, 402 или 403 — все **оставшиеся** запросы в batch помечаются как FAILED без отправки. Повторять с невалидным ключом или пустым балансом бессмысленно.

### Восстановление при рестарте

При запуске сервера `GenerationStartupCleaner` находит незавершённые batch (запросы со статусом RUNNING или PENDING) и вызывает `resumeBatchAsync()` для каждого. Обработка продолжается в фоне — уже завершённые запросы пропускаются, оставшиеся обрабатываются заново.

## 13. Хранение изображений

### Сохранение на диск

Выполняется `ImageStorageService.save(byte[])`:

1. **Валидация:** проверяется размер (≥ 1024 байт) и magic bytes (PNG: `89 50 4E 47`, JPEG: `FF D8 FF`, WebP: `RIFF....WEBP`)
2. **Расширение:** определяется по magic bytes → `.png` / `.jpg` / `.webp`
3. **Имя файла:** `UUID.randomUUID() + расширение`
4. **Запись:** `Files.write(path, bytes)`
5. **Возвращает:** относительный путь для записи в БД (`generated-imgs/a1b2c3d4.png`)

### Структура

```
ai-img-back/
├── generated-imgs/           ← папка с картинками
│   ├── a1b2c3d4-...-e5f6.png
│   ├── g7h8i9j0-...-k1l2.webp
│   └── ...
```

Папка создаётся автоматически при запуске сервера. Поддерживаемые форматы: **PNG, JPEG, WebP**.

**Примерный расход диска:** одна картинка ≈ 200–500 КБ. 10 000 картинок ≈ 2–5 ГБ.

## 15. Узкие места

### Rate limits провайдеров

| Провайдер    | Лимит (Tier 2)        | Что происходит при превышении |
| ------------ | --------------------- | ----------------------------- |
| OpenAI       | ~15 RPM (images)      | 429 Too Many Requests         |
| Stability AI | 150 запросов / 10 сек | 429 + lockout ~60 сек         |

При batch 20 изображений на OpenAI (~15 RPM лимит) система начнёт получать 429 уже на 16-м запросе. Retry смягчает это, но увеличивает общее время ожидания.

### Буферизация в памяти

Каждый вызов `generate()` возвращает `byte[]` (200–500 КБ). Байты не стримятся на диск — хранятся в памяти до сохранения. OpenAI дополнительно хранит base64-строку в памяти (~2–5 MB на картинку) на время декодирования.

**Почему так:** `RestTemplate` возвращает тело ответа целиком как `byte[]` / `String`. Для стриминга на диск нужно перейти на низкоуровневый `ClientHttpResponse.getBody()` → `InputStream` → `Files.copy()`, либо использовать `WebClient` (reactive). Текущий подход проще в коде и достаточен при последовательной генерации — в памяти находится максимум одна картинка за раз. Проблема возникнет при параллельной генерации несколькими пользователями одновременно.

### Retry без jitter

Retry-стратегия использует фиксированную формулу `2^attempt` секунд без случайного смещения (jitter). При 429 от OpenAI повторная попытка через 2 секунды может снова попасть в лимит. Заголовок `Retry-After` из ответа провайдера не учитывается.

**Почему так:** для одного пользователя фиксированный backoff работает — запросы идут последовательно и паузы между ними дают провайдеру передышку. Jitter нужен, когда несколько клиентов ретраят одновременно и их паузы совпадают («thundering herd»). При одном серверном ключе и нескольких пользователях это может произойти — все retry попадут в одну и ту же секунду и снова получат 429.

### Retry внутри HTTP-запроса

Все retry происходят внутри одного HTTP-запроса от клиента. Если 20 запросов и каждый делает 3 retry — клиент ждёт `20 × 3 × 8 = 480` сек только на повторах.

**Почему так:** генерация выполняется в синхронном `POST /generations` — клиент отправил запрос и ждёт ответ со всеми результатами. Retry встроен в цикл обработки, чтобы не терять запрос при временных сбоях. Альтернатива — вынести генерацию в фоновую задачу (клиент получает `batchId`, потом поллит статус), но это требует переделки API и клиента.

### Хранилище

Хранилище растёт бесконтрольно. Нет механизма очистки старых файлов, лимита на общий размер, уведомления при заполнении диска.
